2.1.2 Use boards to manage project status

todo -> in progress -> done or cancelled (as archives)

Boards:

  • ! To Do: projects in plan
  • ! In Progress: projects in progress
  • ! Done: archived accomplished projects
  • ! Cancelled: archived incomplete projects
  • ! all: all projects
  • individual boards for specific projects

Sprints could only be created in 'In Progress' board.

2.2 Hierarchy

(Refer to book "team tools" 4.7)

2.2.1 Epic

~= 系统需求确定之前的需求定义阶段

特征：

  • 很大的需求
  • 跨部门的课题
  • 和工程师不相关的事情居多
  • 主要目的：确定优先顺序（priority）
    • 多以一览表（checklist）的形式呈现
    • 开始时多处于模糊的状态 -》不适合采用固定的形式，需要能灵活修改的形式, 方便多人分享和协作 -》多采用电子表格工具（e.g. google docs）

示例：

  • 构建开拓新市场用的系统
  • 为每个客户准备信息中心（dashboard）

2.2.2 Story (Jira's Task)

~= 具体的功能要求或规格

特征：

  • 在 scrum 中表示为用户带来的价值
  • 多由产品负责人编写（product owner in scrum）
  • 需要进行状态管理（versioning）【？？？】
    • 以团队为分界线，如果涉及多个部门，可以采用电子表格工具进行管理
    • 各部门可以考虑使用 issue tracking system 进行管理
  • 最好开始和代码建立关联
  • 负责人：多人或一个人

编写者：根据实际情况，可以由非工程师编写，也可以由 developer 编写

示例：

  • 在顾客登录时显示定制欢迎信息
  • 统计每个用户 ID 的访问情况，支持在 dashboard 阅览和下载

2.2.3 Task (Jira's Sub-task)

即作业，和价值没有直接关系，但如果不完成 task 的话 story 的完成就无从谈起。

特征：

  • 具体的作业内容
  • story 下关联着多个 task
  • 只有一个负责任
  • 根据需要确定是否需要和代码进行关联
  • Managed using issue tracking system

示例：

  • 初始化系统
  • 写单元测试

2.2.4 Bug

即 QA 部门发来的 bug 报告、故障的处理以及顾客的咨询等等。

特征：

  • 程度上和 task 相近
  • 通常由单个开发人员负责
  • 需要关联代码
  • 应作为 Issue tracking system 的管理对象
  • 如果 bug 数量很多，可以将 bug 管理独立出来（复杂度增加，不推荐）-》尽可能在同一个 issue tracking system 下管理 story 和 bug。

4 Deployment

4.1 Capistrano

(http://capistranorb.com/documentation/overview/what-is-capistrano/)

Capistrano is a remote server automation tool.

It supports the scripting and execution of arbitrary tasks, and includes a set of sane-default deployment workflows.

Capistrano can be used to:

  • Reliably deploy web application to any number of machines simultaneously, in sequence or as a rolling set
  • To automate audits of any number of machines (checking login logs, enumerating uptimes, and/or applying security patches)
  • To script arbitrary workflows over SSH
  • To automate common tasks in software teams.
  • To drive infrastructure provisioning tools such as chef-solo, Ansible or similar.
  • Capistrano is also very scriptable, and can be integrated with any other Ruby software to form part of a larger tool.
