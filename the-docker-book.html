1.3 docker run: run container

docker run -i -t ubuntu /bin/bash

docker run -i -t alpine /bin/sh

1.3.1 –name: name a container

docker run --name myubuntu -i -t ubuntu /bin/bash

1.3.2 -d: daemonize a container

Types of containers:

  • interactive container
  • daemonized container 
docker run --name myubuntu -d ubuntu /bin/sh -c "while true; do echo hello world; sleep 1; done"

Tip: use docker attach or docker logs to see the output.

1.3.3 –restart: restart a container automatically

# always restart
docker run --restart=always --name myubuntu -d ubuntu /bin/sh -c "while true; do echo hello world; sleep 1; done"

# restart on failure, retry 5 times
docker run --restart=on-failue:5 --name myubuntu -d ubuntu /bin/sh -c "while true; do echo hello world; sleep 1; done"

1.4 docker ps: list containers

List running containers: 

docker ps

List all containers: 

docker ps -a

Container still exists even if you exit it (using exit command).

1.5 docker start + docker attach: start a stopped container and attach to it

docker start my-alpine
docker attach my-alpine

Tip: You may need to press Enter key to get in the shell.

1.6 docker logs: peek into the container

docker logs myubuntu

# follow output
docker logs -f myubuntu

# with timestamp
docker logs -ft myubuntu

# show the last 10 lines
docker logs --tail 10 myubuntu

# follow the latest output
docker logs --tail 0 -f myubuntu

1.7 docker top: list processes running in container

docker top myubuntu

1.8 docker exec: start new process in container

# start shell in a daemonized container
docker exec -it myubuntu bash

# create a new file in a daemonized container
docker exec myubuntu touch /etc/sample.txt

# start a daemonized container and create a new file in it
docker exec -d myubuntu touch /etc/sample.txt

1.9 docker stop: stop a daemonized container

Tip: send SIGTERM signal.

2 4. 使用 docker 镜像和仓库

2.1 What is docker image?

Made up of multiple layer of file systems (based on 'union mount' technology):

  1. bootfs (unmounted after the container is loaded)
  2. rootfs (read-only)
  3. more read-only file systems
  4. read-write file system (where app is running)

Resulted filesystem -> image!

Image -> parent image -> … -> base image [Refer to Fig. 4-1]

创建新容器时，Docker 会构造一个 image stack，并在 stack 顶端添加一个 read-write layer（copy on write）。其中的每个 read-only layer 以后永远不会变化。

容器可以修改，有自己的状态，可以启动和停止 + image-layering framework = 快速构建 image 并运行包含 app 和 service 的容器

2.1.1 Copy on write

如果修改一个文件，这个文件首先从下面的 read-only layer 复制到该 read-write layer。该文件的只读版本依然存在，但是已经被 read-write layer 中的该文件副本所隐藏。

2.5.2 Use docker build & Dockerfile

Dockerfile's syntax is based on DSL grammar.

Build environment (context or build context): 用于保存 Dockerfile 的目录。–》Docker 会在构建镜像时将 context 和该 context 中的文件和目录上传到 Docker daemon，这样 Docker daemon 就能直接访问想要在镜像中存储的任何代码、文件或其他数据。

  1. Dockerfile
    • 指令必须大写，后跟参数
    • 顺序执行
    • 每条指令都会创建一个新的镜像层并进行提交
    1. 执行流程
      1. Docker 从基础映像运行一个容器
      2. 执行指令，修改容器
      3. 执行类似于 docker commit 的操作，提交新镜像层
      4. Docker 再基于新镜像层运行一个新容器
      5. 以此类推

      Tip: 如果中间发生错误，仍然能得到新镜像 –》特别有利于调试！看看为什么失败

      Note: 默认情况下，RUN指令使用 /bin/sh -c 来执行，如需避免使用 sh（例如命令行篡改副作用），可以使用 exec 格式： 

      RUN["apt-get", "install", "-y", "nginx"]

      Note: EXPOSE 指令不会让 Docker 自动打开端口，必须在 docker run 时指定需要打开哪些端口。可以使用多个 EXPOSE 指令公开多个端口。

  2. docker build
    docker build -t="jamtur01/static_web" .

docker build -t="jamtur01/static_web:v1" .

docker build -t="jamtur01/static_web:v1" git@github.com:jamtur01/docker-static_web

    Explanation:

    • . 告诉 Docker 在本地目录里寻找 Dockerfile 文件
    • 最后一个关于 git 的命令，Docker 假设这个 Git 仓库的更目录下存在 Dockerfile 文件
    • 如果没有指定标签，Docker 将会自动为镜像设置 latest 标签。
    1. .dockerignore 文件

      如果在 build context 根目录下存在 .dockerignore 文件，将会按行过滤文件（非常类似于 .gitignore），用于设置哪些文件不会被上传到 build context【也就是说不会被发送给 Docker daemon】

      匹配规则使用 Go 语言的 filepath.

    2. Debug

      构建失败时将会输出错误信息。

      使用 docker run 运行失败中断时生成的新镜像，在其中找出问题后修改 Dockerfile。

    3. Cache
      • Docker 会将 Dockerfile 里每一步构建的结果镜像看做缓存 –> 节省时间；复用模板（在 Dockerfile 顶部增加包仓库或更新包）
      • 如果修改了指令，Docker 会从第一条发生了变化的指令开始再次构建

      如果要略过缓存，可以使用 –no-cache option： 

      docker build --no-cache -t="jamtur-1/static_web:v1"

      Best Practice: 在 Dockerfile 顶部使用相同的指令集模板，使用 ENV 指令设置 REFRESHAT 环境变量，通过修改 ENV 指令里的日期重置缓存 

      FROM ubuntu:14.04
MAINTAINER James Turnbull "james@example.com"
ENV REFRESHED_AT 2014-07-01
RUN apt-get -qq update
...
  3. review new image
    docker images jamtur01/static_web

docker history <image_id>

    #check docker port mapping docker ps -l docker port <containerid>

  4. run a container from an image
    docker run -d -p 80 --name static_web jamtur01/static_web nginx -g "daemon off;"

    Explanation:

    • -d option: tell Docker to run on background in detached mode, suited for long-run process like nginx
    • =nginx -g "daemon off;"=: tell nginx to run in foreground 【重要，否则 container 将中止，参见 nginx.conf】
    • -p: control Docker to publish which ports to the host
    1. 在 host 上分配端口

      Tip: 可以在端口绑定时使用 /udp 来指定 UDP 端口 两种方式

      1. 在 host 上随机选择一个 49000~49900 的端口

        映射到 container 指定的端口上，代码如上所示。

      2. 在 host 上指定一个具体的端口号

        映射到 container 指定的端口上

        1. 将 container 的 80 端口绑定到 host 的 80 端口

          【格式：host:container。需要非常小心，这种做法将会限制 docker 的灵活性】： 

          docker run -d -p 80:80 --name static_web jamtur01/static_web nginx -g "daemon off;"
        2. 将端口绑定限制在特定网络接口（即 IP 地址）上
          docker run -d -p 127.0.0.1:80:80 --name static_web jamtur01/static_web nginx -g "daemon off;"
        3. 将端口绑定限制在特定 IP 的随机端口上
          docker run -d -p 127.0.0.1::80 --name static_web jamtur01/static_web nginx -g "daemon off;"
    2. -P option: simpler solution

      对外公开 Dockerfile 中 EXPOSE 指令设置的所有端口，绑定到 host 的随机端口 

      docker run -d -P --name static_web jamtur01/static_web nginx -g "daemon off;"
  5. Dockerfile 指令

    Refer to https://docs.docker.com/engine/reference/builder/.

    1. RUN
    2. EXPOSE
    3. CMD
      • 指定 container 启动时需要运行的命令
      • 可以为要运行的命令指定参数： 
      CMD["bin/bash", "-l"]
      • 推荐使用数组语法（避免默认使用 /bin/sh -c 带来的副作用）
      • docker run 命令可以覆盖 CMD 指令
      • Dockerfile 中如果有多条 CMD，那么只有最后一条 CMD 指令会被执行（如果想要在启动容器时运行多个进程或多条命令，可以考虑使用supervisor 这样的服务管理工具【？？？】）
    4. ENTRYPOINT

      与 RUN 的区别：

      • docker run 会覆盖 CMD，ENTRYPOINT 不会【不容易被覆盖】，docker run 中的任何参数都会被当做参数传递给 ENTRYPOINT 指令中指定的命令
      • 可以使用 docker run 的 --entrypoint 标志覆盖 ENTRYPOINT 指令

      *Best Practice*：可以组合使用 ENTRYPOINT 和 CMD 完成一些巧妙的工作 

      ENTRYPOINT ["usr/sbin/nginx"]
CMD ["-h"]

      Explanation:

      • 如果在启动 container 时不指定任何参数，则 CMD 中指定的 -h 将会被传递给 nginx，显示 nginx 的帮助信息
      • 这样可以构建一个镜像，既可以运行一个默认的命令，也支持通过 docker run 命令行为该命令指定可覆盖的选项或标志
    5. WORKDIR
      • 在 container 内部设置一个工作目录，ENTRYPOINT 和 CMD 指定的程序会在此目录下运行
      • 既可以使用该指令为 Dockerfile 中后续指令设置工作目录，也可以为最终的 container 设置工作目录，例如： 
      WORKDIR /opt/webapp/db
RUN bundle install
WORKDIR /opt/webapp
ENTRYPOINT ["rackup"]

      Tip: 可以通过 -w 标志在运行时覆盖工作目录： 

      docker run -it -w /var/log ubuntu pwd
    6. ENV
      ENV TARGET_DIR /opt/app
WORKDIR $TARGET_DIR
      • 在 image building 过程中设置 env var
      • 新的 env var 可以在后续的其他指令中使用
      • 这些 env var 会被持久保存到从 image 创造的任何 container 中

      Tip: 在命令行可以直接定义 env var 

      RVM_PATH=/home/rvm/ gem install unicorn

      Tip: 可以使用 docker run 的 -e 标志来传递 env var，只在运行时有效！

    7. USER
      USER nginx
USER user
USER user:group
USER uid:gid
USER user:gid
...
      • 指定什么样的用户运行指定 image
      • 可以指定 user name 或 UID 以及 group 或 GID，甚至是两者的组合
      • 如果没有指定 user，默认是 root
    8. VOLUME
      VOLUMN ["/opt/project"]

VOLUMN ["/opt/project", "/data"]
      • 向基于 image 创建的 container 添加卷
      • 卷：可以存在于一个或多个容器内的特定目录，这个目录可以绕过联合文件系统(Union File System)，提供如下功能用于共享数据或对数据进行持久化：
        • 可以在 container 间共享和重用
        • container 不是必须和其他 container 共享卷
        • 对卷的修改将立即生效
        • 对卷的修改不会对 image 产生影响
        • 卷会一直存在，知道没有任何 container 再使用它

      Tip: 卷功能让我们可以将数据（例如源代码）、数据库或其他内容 添加 到 image 中但不会把这些内容 提交 到 image，并且允许在多个 container 间共享这些内容。–》可以利用此功能来测试容器和内容的 app 代码、管理日志、处理容器内部的数据等。

    9. ADD
      ADD software.lic /opt/application/software.lic

ADD http://wordpress.org/latest.zip /root/wordpress.zip

ADD latest.tar.gz /var/www/wordpress/
      • 将 build context 下指定的文件和睦路复制到 image 中
      • 参数：源文件位置，目的文件位置。源文件位置可以是 URL、build context 内的文件或目录，不能对 build context 之外的文件进行 ADD 操作。
      • ADD 文件时，通过目的参数末尾字符是否为 "/" 来判断是源文件是目录还是文件
      • 源文件是 tar archive（后缀名是 gzip、bzip2、xz）时，Docker 会把文件自动解开。–> 结果是目的目录原来的内容加上归档文件中的内容。不会覆盖目的位置中的文件。
      • ADD 会自动创建不存在的目录。新创建的文件和睦路模式为 0755，UID 和 GID 都是 0.

      Note: ADD 指令会让 build cache 无效！

    10. COPY
      COPY conf.d/ /etc/apach2/
      • 非常类似于 ADD，不同之处：COPY 只关心在 build context 中复制本地文件，不做 extraction 和 decompression
    11. ONBUILD
      ONBUILD ADD . /app/src
ONBUILD RUN cd /app/src && make
      • 为 image 添加触发器（trigger），当这个 image 被其他 image 用作 basic image 时，该 image 中的 trigger 将会被执行
      • 示例用途：
        • image 需要从某个未准备好的位置添加源代码
        • 需要执行特定于将要构建的 image 环境的 build script
      • ONBUILD 会在 build 过程中插入新指令（可以是任何构建指令），可以认为这些指令被加入到新 image 的 FROM 指令后面【紧跟 FROM 指令！】
      • 可以用 docker inspect 查看

      例子：构建一个 Apache2 image（jamtur01/apache2） 

      FROM ubuntu:14.04
MAINTAINER ...
RUN apt-get update
RUN apt-get install -y apache2
ENV APACHE_RUN_USER www-data
ENV APACHE_RUN_GROUP www-data
ENV APACHE_LOG_DIR /var/log/apache2
ONBUILD ADD . /var/www/
EXPOSE 80
ENTRYPOINT ["/usr/sbin/apache2"]
CMD ["-D", "FOREGROUND"]

      Explanation:

      • build 这个 image 时 ONBUILD 将会被执行
      • build 基于这个 image 的新 image 时 ONBUILD 也将会被执行
      • 【也就是说，保证 ONBUILD 语句里的指令在 build 新 image 时也会被执行 –》动态更新此处指令开始的所有层！–》可以作为模板使用！– 每次都将本地源代码添加到 image，支持为不同的 app 进行一些特定设置！】
      • 只能被集成一次：子 image 内执行，孙子 image 内不会被执行
      • 有几条指令不能用在 ONBUILD 语句中【防止递归调用】：
        • FROM
        • MAINTAINER
        • ONBUILD

3.1.2 Get nginx configurations under ./nginx/

  1. global.conf
    server {
        listen          0.0.0.0:80;
        server_name     _;

        root            /var/www/html/website;
        index           index.html index.htm;

        access_log      /var/log/nginx/default_access.log;
        error_log       /var/log/nginx/default_error.log;
}
    • listen to port 80
    • set root path to "/var/www/html/website"
  2. nginx.conf
    user www-data;
worker_processes 4;
pid /run/nginx.pid;
daemon off;

events {  }

http {
  sendfile on;
  tcp_nopush on;
  tcp_nodelay on;
  keepalive_timeout 65;
  types_hash_max_size 2048;
  include /etc/nginx/mime.types;
  default_type application/octet-stream;
  access_log /var/log/nginx/access.log;
  error_log /var/log/nginx/error.log;
  gzip on;
  gzip_disable "msie6";
  include /etc/nginx/conf.d/*.conf;
}
    • set nginx to non-daemon mode to make it running in Docker container - daemon:off; 阻止 nginx 进入后台

    Note: 要想保持 Docker container 的活跃状态，需要其中运行的 process 不能中断。因为默认情况下，nginx 以守护进程方式启动，这会导致 container 只是短暂运行，在 daemon 被 fork 启动后，发起 daemon 的原始 process 就会退出，这时 container 将停止运行。

    • 如果 container path 不存在会自动创建一个
    • 对卷的修改会直接生效
    • 提交或创建 image 时，volume 不被包含在 image 里
    • 可以在 container path 后加上 rw 或 ro 指定读写状态，例如 /var/www/html/website:ro 指只读状态

Note: 使用卷的原因（不想把 app 或代码 build 到 image 里）

  • 希望同时 开发 和 测试 代码
  • 代码改动很频繁，不想经常 build image
  • 希望多个 container 共享代码

4.9 docker create

(https://docs.docker.com/engine/reference/commandline/create/)

The docker create command creates a writeable container layer over the specified image and prepares it for running the specified command. The container ID is then printed to STDOUT. This is similar to docker run -d except the container is never started. You can then use the docker start <containerid> command to start the container at any point.

This is useful when you want to set up a container configuration ahead of time so that it is ready to start when you need it. The initial status of the new container is created.

Example: 

$ docker create -t -i fedora bash
6d8af538ec541dd581ebc2a24153a28329acb5268abe5ef868c1f1a261221752
$ docker start -a -i 6d8af538ec5
bash-4.2#
