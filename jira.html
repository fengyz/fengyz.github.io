2.2 Creating a Board

We need a board so that we can view and work with issues.

A board displays issues from one or more projects.

two types of boards:

  • Scrum boards: for teams that plan their work in sprints.
  • Kanban boards: for teams that focus on managing and constraining their work-in-progress. Because work is not planned in advance using discrete time periods or versions, Kanban boards do not have a Backlog screen.

Note Before you begin, please note that you will need the JIRA 'Create Shared Objects' global permission before you can create a board. If you don't have this permission you can still create a board, but it will not be visible to other people as its filter will not be shared.

You can either copy an existing board, or create a completely new one.

We can create a board with sample data.

Note: creating a project from JIRA Agile requires the 'Browse Projects' permission in the Default Permission Scheme.

JIRA offers boards based on filters. Refer to 'Example JQL for your board's Saved Filter' at this page.

If you would like to create a board that is /similar to one you are already using, you can simply create a copy.

2.3.2 Using the Backlog

This is whare you plan your sprints. Click Start Sprint to move a ready-to-go sprint into the Active sprints.

In the Backlog of the board you can:

  • Prioritise the backlog. (Tip: right-click select issues.)
  • Estimate Stories. J/K to move through issues. Plug in your estimates or story points as you go. (Note: by default, the Story Points field is only available to issues of type 'Story' or 'Epic'. You can change this as described in JIRA Agile JIRA configuration)
  • Create Sub-Tasks. To break a story (issue) down into implementable chunks.
  • Create new Issues.
  • Organise via Epics. Groupe related stories into an epic.
  • Identify the workload for specialists. Click the "…" on the right side of the avatars for users.
  • Plan Versions. Assign issues to upcoming versions. On the Versions panel, you can assign issues to versions via DnD.
  • Plan, and Plan Again. When you're happy with the stories for the iteration, start a sprint and the stories will move into the Active sprints. You can still plan subsequent iterations in the Backlog (click Add Sprint), but you won't be able to start them until the active iteration is completed. You can, however, DnD and issue in the Backlog onto the active sprint.
  1. Using Quick Filters

    Refines the collection of issues displayed on your board. Appear immediately below the board name in the Backlog and Active sprints.

    Refer to configuring Quick Filters.

2.3.3 Using the Active sprints or Kanban board

Active sprints (Scrum boards) is where you transition your issues through a series of columns (statuses).

In the Active sprints/Kanban board you can:

  • Work on Issues. Choose the highest ranked issues in the 'To Do' column and drag them to the 'In progress' column. And as you complete each issue in it, drag it to the 'Done' column. (Tip: select one/more issues, then right-click)
  • Release a Version. Click Release to create your new version and mark it as 'released' throughout JIRA. (See Releasing a Version)

Note:

  • each column maps to one/more JIRA statues.
  • resolved issues are shown in strike-through font.
  • 'Days in column' indicator
  • the board will be refreshed every 30 seconds, if the browser window does not currently have focus. If you have focus on the window, it will show a message asking you to refresh.

3.1 公司推行jira，要求工程师每天log work，监督每个工程师有没有做满8小时的工作，想请教下各位国外IT界大牛是否有必要，是否有这样做的优秀的IT公司？

(link @zhihu)

之前在一家美国公司做的时候推行过这样的管理制度。从管理的角度来说，记录工作日志这种事情毫无违和感，如果不需要你记录工作日志，而让你写工作日报，难道你就感觉舒服一些了？这是两条管理规则一起推动的结果，所以工程师认为记录工作日志是额外的工作。我们公司当时是，先要求提交详细的工作日报，精确到问题、小时，这样执行了两个多月，员工最初怨声载道，后来逐渐麻木习惯，然后告诉大家一个好消息，在JIRA里直接记录工作时间，一天只要凑7个小时就可以了！这样，只要谁还觉得在JIRA里记录工作麻烦的，就让丫自己去写工作日报去。JIRA进行时间管理，通过三个方面提高大家的工作效率：总体工作时间要求：每周40小时；每问题工作时间要求：平均问题解决时间，疑难问题识别问题对工作内容的覆盖率：任何超过一小时的工作都应创建问题第一条要求工程师事无巨细都必须上报，第二条要求上报内容真实可靠。硬性要求总体时间后，工程师必须想尽办法凑时间进去，而第二条，例如我当年，要求每问题平均解决时间为3小时，凡是超过3小时的必须写明问题原因。这样，工程师不敢在一个任务上面谎报大量时间，同时，靠这个办法发现了大量平时掩盖的问题，集中解决后效率提高不是一星半点。配合 Fisheye 插件集成SCM工具，你可以看到，一个号称花费了40小时的任务，到底有多少代码变更。直观的就可以识别出工作效率问题。这种证据是可以用在辞退工程师上的。当年实施下来，有数名工程师离职，坦白说，都是滥竽充数的。有的离职时直接跟我说，本来就是图个清闲，现在忙成这个样，不干了；有的是因为，干了几十个小时的任务，只有个位数的代码变更。我们不是什么研究算法的团队，这种事儿就是他娘的混。而真正有效率的工程师，勤恳的工程师，马上就可以看出来。管理者通过JIRA进行管理也同样是要求很高的。最后一条是套在管理者头顶的魔咒。管理者不能是浑浑噩噩的混子，而是应该对自己的团队工作范围有明确的定义，对团队工作进度有良好的管理，这样才能将所有的工作内容都使用不同类型的问题覆盖。经常有这种实施失败的案例，最大的问题并非工程师没有执行，而是管理者水平太烂，和稀泥，于是就流于形式。不说别的，如果问题能够对工作内容实现完整覆盖，很容易回答的一个问题是，团队的生产时间占总体比例是多少？如果回答不了这个问题，那是个什么鸡巴管理者就一眼看出来了。Atlassian是什么样的公司，JIRA以及他们家的一大堆产品设计如何，在行业内的地位和水平都是无须讳言的。那么，*TEMPO* 插件火成这个样子，难道老外都咸的蛋疼？

4.2 Using Smart Commits

(https://confluence.atlassian.com/fisheye/using-smart-commits-298976812.html)

The basic syntax for a Smart Commit message is: 

<ignored text> <ISSUE_KEY> <ignored text> #<COMMAND> <optional COMMAND_ARGUMENTS>

Any text between the issue key and the command is ignored.

There are three commands you can use in your Smart Commit messages:

4.2.1 comment

<ignored text> <ISSUE_KEY> <ignored text> #comment <comment_string>

JRA-34 #comment corrected indent issue

4.2.2 time

<ignored text> <ISSUE_KEY> <ignored text> #time <value>w <value>d <value>h <value>m <comment_string>

JRA-34 #time 1w 2d 4h 30m Total work logged

4.2.3 transition

<ignored text> <ISSUE_KEY> <ignored text> #<transition_name> <comment_string>

JRA-090 #close Fixed this today
