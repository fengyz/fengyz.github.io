  • server-side javascript container
  • web app framework
  • database server: RDBMS, NoSQL database (document database, key/value pair database, e.g. MongoDB)

  • Mongo -> database server
  • Express -> web app framework

  • Node -> server-side javascript container

    var express = require('express');
var app = express();
app.set('port', process.env.PORT || 3000);
// custom 404 page
app.use(function(req, res){
  res.type('text/plain');
  res.status(404);
  res.send('404 - Not Found');
});
// custom 500 page
app.use(function(err, req, res, next){
  console.error(err.stack);
  res.type('text/plain');
  res.status(500);
  res.send('500 - Server Error');
});
app.listen(app.get('port'), function(){
  console.log( 'Express started on http://localhost:' +
              app.get('port') + '; press Ctrl-C to terminate.' );
});
  5. 5. 开始运行：node meadowlark.js，或使用 supervisor meadowlark.js

6.1 QA: Is It Worth It?

在 web 开发中，质量可以分为四个维度：

  • Reach（影响力）：影响力直接关系到盈利能力，SEO 非常总要，需要在 QA 计划里包含 SEO。
  • Functionality（功能）：一旦用户访问了你的网站或服务，功能性就直接关系到是否能够留住用户 – 实际是否与广告相符。可以实现对功能的测试自动化。
  • Usability（可用性）：功能关系功能是否正确实现，而可用性则用来衡量人-计算机交互（HCI），基本问题是“它容易使用吗？”。通常不能进行自动化测试。
  • Aesthetics（美学）：最主观的维度，与开发工作距离最远。但也需要在 QA 计划里包含美学方面的定期 review（展现给目标客户，看是否过时或是否获得预期的反应）。要注意美学观念经常随时间变化，而且不同用户的美学偏好也有所不同。

6.3 测试的种类

两大类：

  • 单元测试（unit testing）：细粒度（fine-grained），测试单个模块以确定它们是否正常工作。通常更有用，适合于测试逻辑（也能测试呈现部分的代码）。
  • 集成测试（integration testing）：包括了系统测试（system testing），测试多个组件之间的交互过程，甚至是整个系统。对于逻辑和呈现均有用。

6.4 QA 技术一览

  • Page tesing：测试单个页面的呈现以及前段功能，关系到单元测试和集成测试。使用 Mocha 。
  • Crosspage testing：测试需要从一个页面导航到其他页面的功能。例如电子商务网站的 checkout 流程。通常涉及多个组件，因此属于集成测试。使用 Zombie.js 。
  • Logic testing：对逻辑域（logic domain）进行单元测试及集成测试，只测试 JavaScript，与呈现部分完全脱离。
  • Linting【ME：掉毛？】：不是为了找出错误，而是为了找出可能的错误。其基本概念是找出可能发生错误的部分，或者将来可能导致出错的脆弱环节。使用 JSHint 。
  • Link checking：确定网站上是否有 broken link。最好及早在 QA 里包含这项检查。属于单元测试（某个链接有效还是无效）。使用 LinkChecker 。

7.1 HTTP Request Methods

= HTTP verbs，最常见的是 GET 和 POST。

输入 URL 或点击链接时，browser 发出 HTTP GET 请求，其中重要的信息部分是 URL path 和 querystring。

App 处理请求基于 method + path + querystring。

Browser 一般只使用 GET 和 POST，除非使用了 AJAX。Web services 使用的 HTTP verbs 更多一些，例如用 DELETE 请求删除某个东西。

用 Node 和 Express 可以完全控制我们要回应哪些 HTTP methods，在 express 里，我们通常针对某个特定的 method 编写 handler。

7.2 Request Headers

Browser 向 server 发送请求时，不仅仅发送 URL，还有许多其他信息，都包含在 request header 里，包括语言偏好、user agent（browser/os/hardware）等等，这些都可以通过 request 对象的一个属性 headers 访问。我们可以加入下面一个路由来显示这些信息： 

// display request header
app.get('/headers', function (req, res) {
    res.set('Content-Type', 'text/plain');
    var s = '';
    for (var name in req.headers)
        s += name + ': ' + req.headers[name] + '\n';
    res.send(s);
});

7.3 Response Headers

Server 也回送一些不会显示的信息给 browser，包含在 response header 里，包括 metadata 和 server 信息，例如：

  • Content-Type：告诉 browser 传送的内容类型（CSS，HTML，JS 等等），注意 browser 将遵从 server 的指示显示内容（render content），这意味着可以向一个 URL 请求一个 HTML 文件，vice versa。

注意：发送 server 信息可能会给黑客以可乘之机，要关闭 Express 默认的 X-Powered-By header 可以使用代码： 

app.disable('x-powered-by');

可以使用 browser 的开发者工具来查看 response header。

如果特别重视安全，甚至可以伪造 response header。

7.4 Internet 媒体类型

Content-Type header 至关重要，它的格式是一个 Internet Media Type（也可以叫做 content type、MIME type），包括 type、subtype 和可选参数。例如 text/html; charset=UTF-8。

Internet Assigned Numbers Authority 维护了一个官方的 Internet Media Type 列表。

7.5 Request Body

除了 request header，还有 request body。普通的 GET request 没有 request body，但 POST 有。

POST body 的 media type 一般是 application/x-www-form-urlencoded，就是被编码的 name/value 对，用 & 分隔（和 GET 的 querystring 格式一样）。

如果 POST 要支持文件上传，media type 就是更复杂的 multipart/form-data 类型。

AJAX request body 通常使用 application/json 类型。

7.6 参数（parameters）

对任何一个 request，parameter 可能来自于：

  • querystring
  • session（chapter 9）
  • request body
  • 命名的路由参数（named routing parameters，【chapter 14】）

在 Node 里，所有这些参数都能通过 request 对象的 param 方法访问。正是出于这个原因，尽量 避免 使用这个方法 – 如果同名参数在 querystring 及 POST body 里被设置为不同的值，应该使用哪一个？这会导致令人发狂的 bug【ME：想想 PHP 的 $REQUEST 吧】。

不容易产生混淆的解决方案：Node 里将使用专有的属性存放不同类型的 parameters。

7.7 Request 对象

Request 对象通常被传递给一个 callback（参数名通常使用 req 或 request）。它是 Node 核心对象 http.IncomingMessage 的一个实例。Express 又给它添加了许多功能（只有 req.headers 和 req.url 来自于 Node，其他的都来自于 Express）

  • req.params：包含 named route parameters 的一个数组【chapter 14】
  • req.param(name)：返回 route parameter，或 GET/POST parameter，应避免使用
  • req.query：一个对象，包含querystring parameters（name/value 对，也称作 GET parameters）
  • req.body：一个对象，包含 POST parameters。要想使用这个对象，需要能够解析 body 内容类型的 middleware【chapter 10】
  • req.route：当前被匹配路由的信息。通常用于 route debugging。
  • req.cookies/req.signedCookies：对象，包含从客户端传入的 cookie 值【chapter 9】
  • req.headers：从客户端接收的 request headers
  • req.accepts([types])：一个 convenience 方法，用来确定客户端是否支持给定 content type。可选参数 types 可以是一个 MIME type、一个用逗号分隔的列表、或者一个数组。编写公共 API 的人会对这个方法感兴趣。一般假定 browser 总是接受 HTML 类型。
  • req.ip：client 的 IP 地址。
  • req.path：request path（不带 protocol、host、port 和 querystring）
  • req.host：方法。可能会被伪造（spoofed），因此出于安全原因不要使用它。
  • req.xhr：属性，如果 request 来自于 AJAX call 则返回 true。
  • req.protocol：对于我们来说，一般是 http 或 https
  • req.secure：属性，等于 req.protocol = 'https'
  • req.url/req.originalUrl：属性，名字有点用词不当。返回 path 和 querystring。req.url 可以被重写（rewritten）用于内部 routing，req.originalUrl 被设计用于保存原始的 request 和 querystring。
  • req.acceptLanguages：方法。返回 client prefer （human）语言的一个（顺序）数组。

7.8 Response 对象

类似于 request 对象，一般可命名为 res，resp 或 response。是 http.ServerResponse 的一个实例。

最重要的属性和方法：

  • res.status(code)：设置 HTTP status code。Express 默认是 200（OK），还可以设置为 404（Not Found）、500（Server Error），以及其他 status code。对于 redirect（301,302,303,307），优先使用另一个方法 redirect。
  • res.set(name, value)：设置 response header。一般由系统自动设置，不需要人为设置。
  • res.cookie(name, value, [options])，res.clearCookie(name, [options])：设置或清楚客户端的 cookie。需要 middleware 的支持。【chapter 9】
  • res.redirect([status], url)：重定向 browser。默认是 302（Found）。通常尽量少用重定向，除非某个页面永久移动，此时应该使用代码 301（Moved Permanently）。
  • res.send(body），res.send(status, body）：向客户端发送 response，有一个可选的 status code。Express 默认设置 content type 为 text/html。如果要修改类型为 text/plain，需哟啊先调用 res.set('Content-Type', 'text/plain')，然后再使用 res.send。如果 body 里是数组或对象，应该把 response 作为 JSON 发送（设置相应的 content type）。如果我们发送 JSON 类型，建议使用的方法是 res.json。
  • res.json(json)，res.json(status, json)：发送 JSON 数据。
  • res.jsonp(json)，res.jsonp(status, json)：发送 JSONP 数据。
  • res.type(type)：方法，设置 Cotent-Type header，等于 res.set('Content-Type', type)，还可以直接从文件扩展名映射 Internet Media Type，例如 res.type('txt') 将把 Content-Type 设置为 text/plain。有时这个特性很有用（例如自动提供不同类型的多媒体文件），但通常不建议使用，推荐显式地设置 Internet Media Type。
  • res.format(object)：方法，根据 Accept request header 发送不同的内容，主要用于编写 API【chapter 15】，简单的例子：res.format({'text/plain': 'hi there', 'text/html': '<b>hi there</b>'})
  • res.attachment([filename])，res.download(path, [filename], [callback])：方法，把一个名为 Content-Disposition 的 response header 设置为 attachment，这将提示 browser 下载而不是显式 response 的内容。可以指定 filename 提示 browser 应该保存的文件名。res.download 可以直接制定要下载的文件，而 res.attachment 只是设置了 header，还需要自己向 browser 发送内容。
  • res.sendFile(path, [options], [callback])：读取 path 指定的文件并将它的内容发送给客户。尽量少用这个方法，通常使用 static middleware 能更好地完成同样的工作。但是，如果目的是根据某些条件为指向同一个 URL 的请求回送不同的内容，这个方法可能会更方便。
  • res.link(links)：设置 Links response header。这个特殊的 header 很少使用。
  • res.locals, res.render(view, [locals], callback)：res.locals 是一个对象，包含了渲染视图（rendering views）所需要的 default context。res.render 将使用配置好的 templating engine 渲染一个视图。res.render 的 locals 参数将覆盖 res.locals 里的同名成员。注意 res.render 默认的 response code 是 200，如果需要设置不同的 response code 可以使用 res.status。渲染视图部分在【chapter 7】

How the page is reloaded?

9.4.1 首先要考虑用什么 path（action） 处理 form？

  • 常见方案：使用同样的 path 来显示 form 以及处理 form，头一个是 GET，后一个是 POST，此时可以省略 form 的 action
  • 另一个方案：使用不同的 path 处理 form。例如页面使用 /contact，使用 /process-contact 处理 form。在多个 URL 都采用同样的提交 form 时使用这个方案（例如多个页面上都有一个 email signup 对话框）

9.4.2 接下来要考虑的问题是应该向 browser 发回什么样的 response？

  • Direct HTML response：这时如果用户刷新页面将会显示警告信息，还会影响到 bookmarking 和 Back 按钮，因此不推荐使用
  • 302 redirect：常用方案，但属于误用，302 的含义是 Found，应该使用 303（See Other）
  • 303 redirect：回应 from 提交请求的推荐方案。HTTP 1.1. 明确规定在 303 重定向后 browser 应该使用 GET request【ME：何意？原文 The HTTP specification specifically indicates that the browser should use a GET request when following a 303 redirect, regardless of the original method. - 应指对 browser 的要求，browser 在303的情况应该向server发送GET请求】。

9.4.3 因为使用了 303，下一个问题是：重定向应该指向哪里？

最常用的方案：

  • 重定向到专门的 success/failure 页面：
    • 好处：分析页面访问量很方便；容易实现
    • 缺点：必须为每种可能性设置 URL；用户体验不是最好（用户必须自己回到原来的页面或者导航到想去的页面）
  • 重定向到原来的页面，加上一个 flash 消息：对于散布在网站各处的小 form 来说，最佳的用户体验是不要打断用户的浏览过程（navigation flow）。最简单的实现方法是使用 form 里的一个被设置为当前 URL 的隐藏 field，使用 flash message 提供反馈信息。
  • 重定向到新位置，加上一个 flash 消息：大的 form 通常有自己的页面，一旦提交就没有必要停留在原来的页面上。这时，需要猜测用户想要去哪儿并重定向。还是需要一个 flash 消息提供反馈。
  • 如果使用的是 AJAX，推荐使用一个专门的 URL，这个 AJAX handler 应该处理普通的 browser 提交，作为一个保障手段（failsafe）。

10 Chapter 9. Cookings and Sessions

使用 signed cookies 防止 cookies 的内容被恶意篡改。

最愚蠢的行为是运行 cookies 中包含的代码。

XSS（cross-site scripting attacks）的一种技术是使用恶意 js 代码修改 cookies 的内容。使用 signed cookies 可以起到保护作用。

过度使用 cookies（例如在客户端储存过多的数据）将会激怒用户，应避免这种情况，keep your use of cookies to a minimum.

尽量使用 session 来代替 cookies，它更安全，也不用担心会让用户不快。虽然 session 基于 cookies，但 Express 将为我们完成大部分工作。

提示：cookies 的工作原理 – 当 server 希望用户存储一个 cookies 时，它向客户发送一个名为 Set-Cookie 的 header，包含了 name/value 对。当客户端向这个 server 发送请求时，它发送多个 Cookie request header，包含了 cookies 的值。

10.1 Externalizing Credentials

cookie secret 是一个字符串，server 使用它在发送 cookies 之前加密其内容。这是一个随机字符串，不是需要记忆的 password。可以使用基于 xkcd 的随机密码生成器建立 cookie secret。

常见的一种做法是把第三方的 credentials 集中存放在一个文件中，例如 cookie secret、数据库密码、API tokens（twitter/facebook。。。）。这样不仅便于维护，而且也可以避免把 credentials 存入版本控制系统（利用 .gitignore），特别是对于存放在开发源码 repository 上的项目。

我们把 credentials 存放在一个 js 文件 credentials.js 里（也可以使用 XML、JSON）： 

module.exports = {
    cookieSecret: 'chenchenxiaowubanhongduizhiyou5geren',
};

使用时只需加入下面的代码： 

var credentials = require('./credentials.js');

10.2 Express 里的 Cookie   step_by_step guide cookies

设置 cookie 时可以指定下列选项：

  • domain：可以给特定 subdomain 制定 cookie。不能给除了当前服务器 domain 之外的其他 domain 设置 cookie
  • path：控制 cookie 应用的 path。path 末尾包含一个隐含的通配符
  • maxAge：指定客户多久（毫秒单位）之后可以删除 cookie，默认在关闭浏览器时删除。也可以使用 expires 选项，但它的格式太复杂，所以推荐使用 maxAge
  • secure：是否只通过安全连接（HTTPS）发送 cookie。
  • httpOnly：设置为 true 时只有 server 才能修改 cookie，客户端 js 不能修改。预防 XSS 攻击。
  • signed：true 则对 cookie 签名，通过 res.signedCookies 访问而不是 res.cookies。被篡改的 signed cookies 将会被服务器拒绝，cookie 的值将会被重设成原始值。

11 Chapter 10. Middleware

什么是 middleware？

从概念上来说，middleware 是一种封装功能的方法：特别是对发往 app 的 request 进行操作的那些功能。

从实践的角度来说，middleware 就是一个接收三个参数的函数：一个 request 对象，一个 response 对象，还有一个 next 函数。（还有一种接收四个参数的形式，用于错误处理，本章后面将会涉及）

Middleware 在 pipeline 里执行，多个 middleware 执行的顺序很重要。在 Express app 里，使用 app.use 向 pipeline 里加入 middleware。

Express 4.0 开始，middlewares 和 routes 按照连入的顺序被调用，顺序就很清楚。4.0 之前，out of order。

一种通常的做法是在 pipeline 最后连入一个 “catch all” handler，用来处理未能和任何 route 匹配的 request，这个 middleware 通常返回 status code 404（Not Found）。

那么在 pipeline 里，什么时候终结一个 request 呢？答案是 next 函数，如果在某一个 middleware 里没有调用 next()，那么 request 的处理就到此为止。

Express 需要理解或注意的地方：

  • route handlers（常被写作 app.VERB）可以被看做只处理特定 HTTP verb 的 middleware。反过来，middleware 也可以被看做处理所有 HTTP verbs（等同于 app.all，处理任意 HTTP verb） 的 route handler。
  • route handler 需要一个 path 作为第一个参数。如果想要这个 path 匹配任意 route，可以使用 “/*”。Middleware 也可以拿一个 path 作为第一个参数，但它是可选的（如果省略，默认等于“/\*，也就是匹配任意 path”）
  • route handlers 和 middlewares 都接收一个 callback，可以有2/3/4个参数：
    • 2个参数：req 和 res
    • 3个参数：req，res，next 函数
    • 4个参数（变成一个 errorhandling middleware）：error 对象，req，res，next 函数
  • 如果没有调用 next()，pipeline 将被终止。如果没有调用 next()，我们应该向客户发送 response（res.send, res.json, res.render，等等），如果没有这么做，客户端将会挂起直到超时。
  • 如果调用了 next()，通常不建议向客户发送 response。如果这么做了，pipeline 里接下来的 middleware 或 route 将会被执行，但他们向客户端发送的输出将被忽略。

例子： 

app.use(function(req, res, next){
    console.log('processing request for "' + req.url + '"....');
    next();
});
app.use(function(req, res, next){
    console.log('terminating request');
    res.send('thanks for playing!');
    // note that we do NOT call next() here...this terminates the request
});
app.use(function(req, res, next){
    console.log('whoops, i\'ll never get called!');
});

更复杂一点的例子：

11.1 Common Middleware

Express 4 开始，Connect 被移除。Express 唯一保留的 middleware 是 static。这样做让 middleware 可以独立于 Express 进化，Express 也不需要再管理种类繁多的依赖关系。

现在需要解决的问题是“到哪儿去找到这些 middleware？”

  • Connect：我们总是会需要它，因此推荐和 Express 一起安装 Connect（npm install –save connect），然后在 app 里使用它（var connect = require('connect');）
  • basicAuth (app.use(connect.basicAuth)();)：基本访问授权（access authorization），只有最基本的功能，只能和 HTTPS 联合使用（用户名和密码明文传输）
  • body-parser（npm install –save body-parser, app.use(require('body-parser')());）：连接了 json 和 urlencoded。
  • json（参见 body-parser）：解析 JSON 编码的 request body。现在还不是很常见，大多数 API 还使用 application/x-www-form-urlencoded（需要 urlencoded middleware），但使用它将使得 app 更加健壮和符合未来发展趋势。
  • urlencoded（参见 body-parser）：处理 form 和 AJAX request 最常见的方法。
  • multipart（已弃用）：推荐使用 Busboy 或 Formidable 来代替（chapter 8）。
  • compress（app.use(connect.compress);）：用 gzip 压缩 response 数据。应该被及早连入（在其他 middleware 发送 response 之前）。作者唯一推荐在 compress 之前连入的 middleware 是 logging 或 debugging middleware（不发送 response）。
  • cookie-parser（npm install –save cookie-parser, app.use(require('cookie-parser')());）：提供 cookie 支持（chapter 9）。
  • cookie-session（npm install –save cookie-session, app.use(require('cookie-session')());）：提供 cookie-storage session 支持。不推荐使用这个方法。必须在 cookie-parser 之后连入（chapter 9）。
  • express-session（npm install –save express-session, app.use(require('express-session')());）：提供 session ID（保存在 cookie 里）session 支持。默认使用 memory store（不适用于 production 环境），可以配置使用 database store【chapter 9，chapter 13】
  • csurf（npm install –save csurf, app.use(require('csurf')());）：提供针对 cross-site request forgery（CSRF）的保护。因为它使用了 session，因此必须再 express-session 后连入。现在它等同于 connect.csrf middleware。但不是说连接了它就自动获得了保护【chapter 18】
  • directory（app.use(connect.directory());）：支持列出静态文件的目录。不要包含这个 middleware，除非出于特殊的原因要这么做。
  • errorhandler（npm install –save errorhandler, app.use(require('errorhandler')());）：向客户端提供 stack trace 和错误信息。不推荐再 production server 上使用这个 middleware，因为它暴露了实现细节，可能会产生安全或隐私问题。【chapter 20】
  • static-favicon（npm install –save static-favicon, app.use(require('static-favicon')(pathtofavicon));）：不是特别必要。就算没有它，只需要在 static 目录下放一个 favicon.ico 也可以达到同样效果。如果要使用的话，必须及早连入。这个 middleware 可以改善性能，还允许我们指定名字不是 favicon.ico 的文件。
  • morgan（以前的 logger，npm install –save morgan, app.use(require('morgan')());）：提供自动化的 logging 支持 – 记录所有的 requests【chapter 20】
  • method-override（npm install -save method-override, app.use(require('method-override')());）：支持 x-http-method-override request header，这个 header 允许 browser 假装使用 GET/POST 之外的 HTTP 方法，可能对 debugging 有用。只有编写 API 的时候才有用。
  • query：解析 querystring，并将数据存放在 request 对象的 query 属性。由 Express 隐含连入，因此不需要自己做这个工作。
  • response-time（npm install –save response-time, app.use(require('response-time')());）：向 response 里添加 X-Response-Time header，提供 response time（单位毫秒），一般只用于性能微调（performance tuning）。
  • static（app.use(express.static(pathtostaticfiles)());）：支持提供静态（public）文件。可以多次连入，指定不同目录【chapter 16】
  • vhost（npm install –save vhost, var vhost = require('vhost');）：从 apache 借用了 virtual hosts 概念，可以在 Express 里更好地管理 subdomains【chapter 14】

12 Chapter 17. Implementing MVC in Express

MVC：

  • Model is a pure view of your data and logic – it does not concern itself with user interaction at all.
  • Views convey models to the user.
  • Controller accepts user input, manipulates models, and chooses what view(s) to display. — Coordinator would be a better term than controller.

微软提出了一个 MVVM 范式（Model-View-View model）引入了一个很有价值的概念：view model，它是某个 model 的一个 transformation。一个 view model 可以结合多个 model，或者多个 model 的部分内容，或者一个 model 的多个部分。它的价值在于保护了 model – MVC 范式里，很容易用与 views 相关的 transformation & enhancement 污染 model。而 view model 提出了一个解决方案：if you need a view of your data that's only needed for presentation, it belongs in a view model.

作者推荐尽量遵守 MVC 范式，这样可以减少维护问题和技术负债（techincal debt）。

12.1 Models

最重要的组成部分。

重要：不要用 presentation 或用户交互代码污染 models。

理想情况下，model 应该和 persistence layer 完全分离。但在实际中，强行分离两者往往是得不偿失，因为 model 里的 logic 往往严重依赖于 persistence。

12.3 Controller

Controller 负责处理用户交互，基于用户交互选择并显示相应的 views。

听起来像是 request routing – 实际上，二者之间的差别就在于 controller 一般会把相关的功能进行分组。–》 可以把 routing 分组包装成 controller。

例如，一个 customer controller 负责显示和编辑 customer 信息，如下所示建立一个文件 controller/customer.js： 

var Customer = require('../models/customer.js');
var customerViewModel = require('../viewModels/customer.js');
exports = {
    registerRoutes: function(app) {
        app.get('customer:id', this.home);
        app.get('customer:id/preferences', this.preferences);
        app.get('orders:id', this.orders);
        app.post('customer:id/update', this.ajaxUpdate);
    }
    home: function(req, res, next) {
        var customer = Customer.findById(req.params.id);
        if(!customer) return next();
        // pass this }on to 404 handler
        res.render('customer/home', customerViewModel(customer));
    }
    preferences: function(req, res, next) {
        var customer = Customer.findById(req.params.id);
        if(!customer) return next();
        // pass this on to 404 handler
        res.render('customer/preferences', customerViewModel(customer));
    }
    orders: function(req, res, next) {
        var customer = Customer.findById(req.params.id);
        if(!customer) return next();
        // pass this on to 404 handler
        res.render('customer/preferences', customerViewModel(customer));
    }
    ajaxUpdate: function(req, res) {
        var customer = Customer.findById(req.params.id);
        if(!customer) return res.json({ error: 'Invalid ID.'});
        if(req.body.firstName){
            if(typeof req.body.firstName !== 'string' ||
               req.body.firstName.trim() === '')
            return res.json({ error: 'Invalid name.'});
            customer.firstName = req.body.firstName;
        }
        // and so on....
        customer.save();
        return res.json({ success: true });
    }
}

这里我们把 route 管理和实际功能分离开了。

作者认为没有必要把 routing 和 controlle 完全分离开，当 controller 还需要在其他 UI 下复用的话（例如 native app）可以这么做。

14.3.2 处理 Uncaught Exceptions   step_by_step guide exception failover domain

Node 采用了异步机制，因此 uncaught exception 的处理问题就显得特别重要。

看下面这个例子： 

app.get('/fail', function(req, res){
    throw new Error('Nope!');
});

【ME：访问 http://localhost:3000/fail 将跳转到 500 页面，console 将会显示错误信息和 stack trace 信息。】

Express 执行 route handler 时，会把它们包裹在 try/catch 块里，所以实际上不会出现 uncaught exception，这样做一般不会有什么问题 – Express 将在 server 端记录这个 exception，如果没有 500 页面的话，网站访问者会看到一堆难看的 stack dump 信息。

提供自定义错误页面是个好习惯：用户看起来觉得更专业，更重要的是让我们在错误发生时采取行动 – 例如可以在这里给开发团队发送 email。可惜的是，所有这些只有在 Express 能 catch exception 的时候才有用。

看下一个例子： 

app.get('/epic-fail', function(req, res){
    process.nextTick(function(){
        throw new Error('Kaboom!');
    });
});

Server 将会被 拖垮 ！因为 setTimeout 被异步执行，也就是说在 Express 空闲的时候被执行。问题是，当 Node 进入空闲状态并执行这个函数（并抛出异常）时，它已经不再拥有当时它服务 request【ME：匹配这个 path 的 route handler 使用 process.nextTick 调用了这个函数】的上下文，因此只能关闭整个 server，因为现在它处于未定义状态（Node 不知道这个函数的目的，也不知道谁调用了它【ME：route handler】，因此 Node 不知道接下来应该执行哪个函数是正确的）。

注意：process.nextTick 非常类似于调用 setTimeout(0)，但前者更有效率。 这里使用它只是为了示范，我们 不应该 在 server 端使用这个函数。接下来的各种 异步操作 （数据库访问、文件系统访问、网络访问等等）都会遇到前面所说的问题。

发生 uncaught exception 的时候我们其实可以采取行动，但如果 Node 无法确定 app 是否稳定（stable），我们也不能。也就是说，这种情况下唯一能做的事就是尽量优雅地关闭 server 并提供一个故障转移（failover）机制。最常用的 failover 机制就是前面讨论过的 cluster：master 会自动 respawn 死掉的 worker（甚至可以运行只有一个 worker 的 cluster，只是为了它可以作为 failover）。

那么现在的问题是： 如何在碰到 unhandled exception 时尽量优雅地关闭 server？

Node 提供了两种机制来处理这个问题：uncaughtException（未来可能会从 Node 里移除） 和 domains（建议使用）。

Domain 实质上是一个运行环境，可以捕获在其中发生的错误。这个机制允许我们更加灵活地处理错误：不必用一个全局的 uncaught exception handler，我们可以划分多个 domain，这样就可以为可能出错的代码建立新的 domain。

一个好的实践是：在 domain 中处理每个 request，这样我们就可以捕获这个 request 及相关 response 中的任何 uncaught error，优雅地关闭 server。

  1. 1. 在所有 route 定义之前添加一个 middleware：
    // trap any uncaught errors using domains
app.use(function(req, res, next) {
    // create a domain for this request
    var domain = require('domain').create();
    // handle errors on this domain
    domain.on('error', function(err) {
        console.error('DOMAIN ERROR CAUGHT\n', err.stack);
        try {
            // failsafe shutdown in 5
            setTimeout(function() {
                console.error('Failsafe shutdown.');
                process.exit();
            }, 5000);

            // disconnect from the cluster
            var worker = require('cluster').worker;
            if(worker) worker.disconnect();

            // stop taking new requests
            server.close();

            try {
                // attempt to use Express error route
                next(err);
            } catch(err) {
                // if Express error route failed, try plain node response
                console.error('Express error mechanism failed.\n', err.stack);
                res.statusCode = 500;
                res.setHeader('Content-Type', 'text/plain');
                res.end('Server error.');
            }
        } catch(err) {
            console.error('Unable to send 500 response.\n', err.stack);
        }
    });

    // add the request and response objects to the domain
    domain.add(req);
    domain.add(res);
    // execute the rest of the request chain in the domain
    domain.run(next);
});

    解释：

    • 首先建立一个 domain。然后 attach 一个 error handler，当 domain 里发生任何 uncaught 错误时调用这个函数。
    • 我们首先尝试回应任何正在进行（in-progress）中的 request，然后 shutdown server。
    • 基于 error 的性质，有可能无法回应这些 in-progress 的 request，因此要做的第一件事就是设立一个 shutdown 的 deadline。这里我们设定了5秒钟，如果有长时间运行（long-running）的 request 的话，需要更长的时间。
    • 如果当前 app 运行在 cluster 里，那么在设定好 deadline 以后，我们将从 cluster 那里 disconnect worker（worker.disconnect()），这样就可以防止 cluster 继续给这个 worker 分配 request。
    • 然后我们明确告知 server 不再接收新的连接（server.close()）。
    • 最后，我们尝试将发生错误的 request 传递给 error-handling route（next()），以期对 request 做出回应。如果这一步也产生了异常，我们就尝试用文本回应（text/plain）。
    • 如果所有这些努力都失败了，我们就在控制台记录错误信息（此时客户收不到回应，将一直等到 timeout）。
    • 设置了整个 handler 以后，我们要把 req 和 res 对象添加到 domain，这样的话，这些对象包含的方法如果抛出了错误，就都会在 domain 里得到处理。
    • 最后的最后，我们在 domain 的上下文中运行 pipeline 的下一个 middleware【ME：必须，如果改为 next() ，browser 将不显示任何信息，但 server console 输出信息显示发生了异常，server 退出，也就是说 error handler 没起作用。 why？ 】。注意这将在 domain 中运行 pipeline 里的所有 middleware【ME：此处为所有 route 以及 400,500】，因为对 next() 的调用一环接一环。

    作者极力推荐 William Bert's（运行着 SpanishDict 和 Fluencia 两个基于 node 的网站）的文章 The 4 keys to 100% uptime with node.js，他也推荐使用 domain 确保网站的上线时间。

Routing
the mechanism by which requests (as specified by a URL and HTTP method) are routed to the code that handles them.
IA

(no term)
One of the most intelligent and timeless essays on IA is by Tim Berners-Lee (the inventor of the Internet) - http://www.w3.org/Provider/Style/URI.html (written in 1998): "It is the duty of a Webmaster to allocate URIs which you will be able to stand by in 2 years, in 20 years, in 200 years. Suggestions to help you achieve a lasting IA:

  • Never expose techinal details in your URLs
  • Avoid meaningless information in your URLs: like "home" in "homedirections" or "homecontact"
  • Avoid needlessly long URLs: A short URL is bettern than a longer URL, but you should not try to make URLs short at the expense of clarity, or SEO. Be careful with abbreviations.

  • Never use whitespace or untypable characters: Whitespace in a URL will usually just be converted to a plus sign (+), leading to confusion. Don't use any characters other than alphanumeric characters, numbers, dashes, and underscores.

16.6 Organizing Routes

It will be unwiedly to define all of our routes in the main application file. A larger site could have hundreds of routes.

How to organize your routes? It's only limited by your own imagination.

16.6.1 Four guiding principles for deciding how to organize your routes:

  • Use named functions for route handlers: inline, anonymous function is fine for small applications or prototyping, but it will quickly become unwieldy as your website grows.
  • Routes should not be mysterious: For large wbsites, you break the routes out by functional areas.
  • Route organization should be extensible: Whatever method you choose, you should ensure you have room to grow.
  • Don't overlook automatic view-based route handlers: If your site consists of many pages that are static and have fixed URLs, all of your routes will end up looking like this: app.get('/static/thing', function(req, res){ res.render('static/thing'); } . To reduce needless code repetition, consider using an automatic view-based route handler. This approach is described later in this chapter and can be used together with custom routes.

16.9 Automatically Rendering Views

If your website is very content-heavy without a lot of functionality, you don't need to add a route for every view.

Example: You just want to add the file views/foo.handlebars and just magically have it availabe on the route /foo. Add the following middleware right before the 404 handle: 

var autoViews = {};
var fs = require('fs');
app.use(function(req,res,next){
    var path = req.path.toLowerCase();
    // check cache; if it's there, render the view
    if(autoViews[path]) return res.render(autoViews[path]);
    // if it's not in the cache, see if there's
    // a .handlebars file that matches
    if(fs.existsSync(__dirname + '/views' + path + '.handlebars')){
        autoViews[path] = path.replace(/^\//, '');
            return res.render(autoViews[path]);
    }
    // no view found; pass on to 404 handler
    next();
});

Note that regular routes will circumvent this mechanism (because we placed the automatic view handler after all other routes), so if you have a route that renders a different view for the route/foo, that will take precedence.

The approach outlined here offers a great balance between flexibility and effort.

There are some other popular approaches to route organization (you can mix and match techinques, though you run the danger of confusing your architecture).

The two most popular approaches to route organization are namespaced routing and resourceful routing:

  • Namespaced routing is great when you have many routes that all start with the same prefix. There is a Node module called express-namespace that makes this approach easy.
  • Resourceful routing automatically adds routes based on the methods in an object. It can be useful technique if your site logic is naturally object-oriented. The package express-resource is an example of how to implement this style of route organization.

17.2 Our API

We will want the following functionality:

There are many ways to describe our API:

  • Use combinations of HTTP methods and paths to distinguish our API calls, and a mix of querystring and body parameters for passing data.
  • Have different paths (such as /api/attractions/delete) with all the same method.
  • Pass data in a consistent way, e.g. pass all the necessary information for retrieving parameters in the URL instead of using a querystring: GET /api/attractions/:lat/:lng/:radius.

17.3 API Error Reporting

Error reporting in HTTP APIs is usually achieved through HTTP status codes:

  • 200(OK): the request was successful
  • 500(Internal Server Error): the request failed

But in most applications, not everything can (or should be) categorzied coarsely into success or failure.

In general, errors can be grouped into the following categories:

  • Catastrophic errors: Errors that result in an unstable or unknown state for the server. Usually, this is the result of an unhandled exception. The only safe way to recover from a catastrophic error is to restart the server. Ideally, any pending requests would receive a 500 response code, but if the failure is severe enough, the server may not be able to respond at all, and the request will time out.
  • Recoverable server errors: Recoverable errors do not require a server restart. The error is a result of an unexpected error condition on the server (for example, a database connection being unavailable). The problem may be transient or permanent. A 500 response code is appropriate in this situation.
  • Client error: Client errors are a result of the client making the mistake, usually missing or invalid parameters. It isn’t appropriate to use a 500 response code. Everything is working normally, the client just isn’t using the API correctly. You have a couple of options here:
    • You could respond with a status code of 200 and describe the error in the response body.
    • Or you could additionally try to describe the error with an appropriate HTTP status code as followed, the response body should contain an explanation of the specifics of the error. The error message would even contain a link to documentation. Note: if the user requests a list of things, and there's nothing to return, this is not an erro condition, it's appropriate to simply return an empty list.
      • 404 (Not Found)
      • 400 (Bad Request)
      • 401 (Unauthorized)

In our application, we'll be using a combination of HTTP response codes and error messages in the body. This approach is compatible with jQuery, which is an important consideration given the prevalance of API access using jQuery.

17.4 Cross-Origin Resource Sharing (CORS)

How to make the API available to others? This will result in a cross-site HTTP request.

Cross-site HTTP requests have been the subject of many attacks and have therefore been restricted by the same-origin policy, which restricts where scripts can be loaded from. Specifically, the protocol, domain and port must match. This makes it impossible for our API to be used by another site, which is where CORS comes in.

CORS allows you to lift this restriction on a case-by-case basis, even allowing you to list which domains specifically are allowed to access the script.

CORS is implemented through the Access-Control-Allow-Origin header. The easiest way to implement it in an Express application is to use the cors package.

17.4.2 2. Enable CORS for our application:

17.4.3 3. Restrict CORS to specific paths:

18 Chapter 16. Static Content

Static content refers to the resources your app will be serving that don’t change on a per-request basis.

Here are the usual suspects:

  • Multimedia: images, videos, and audio files. It’s quite possible to generate image files on the fly, of course (and video and audio, though that’s far less common), but most multimedia resources are static.
  • CSS: Even if you use an abstracted CSS language like LESS, Sass, or Stylus, at the end of the day, your browser needs plain CSS (It's possible to use uncompiled LESS in a browser with some JavaScript magic. Performance consequences, not recommend) which is a static resource.
  • JavaScript: Client-side JavaScript is considered a static resource. What if there was common code that we wanted to use on the backend and client side? There are ways to solve this problem. But the JavaScript that gets sent to the client is generally static.
  • Binary downloads: This is the catch-all category - any PDFs, ZIP files, installers, and the like.

The HTML doesn't make the list. What about HTML pages that are static? If you have those, it’s fine to treat them as a static resource, but then the URL will end in .html, which isn’t very “modern.” While it is possible to create a route that simply serves a static HTML file without the .html extension, it’s generally easier to create a view (a view doesn’t have to have any dynamic content).

Note that if you are building an API only, there may be no static resources. If that’s the case, you may skip this chapter.

18.1 Performance Considerations

How you handle static resources has a significant impact on the real-world performance of your website, especially if your site is multimedia-heavy.

The two primary performance considerations are

  • reducing the number of requests: MORE CRITICAL, especially for mobile (the overhead of making an HTTP request is significantly higher over a cellular network). It can be accomplished in two ways:
    • combining resources: primarily an architectural and frontend concern - as much as possible, small images should be combined into a single sprite, then use CSS to set the offset and size to display only the portion of the image you want. For creating sprites: SpritePad is highly recommended (it generates the CSS for you as well!).
    • browser caching: helps reduces HTTP requests by storing commonly used static resources in the client's browser. There's a lot you can and should do to enable browser caching for your static resources.
  • reducing content size:
    • lossless techniques: size reduction can be achieved without losing any data. Including minification of JavaScript and CSS, and optimizing PNG images, as well as bundling. – FOCUS of this chapter
    • lossy techniques: size reduction is achieved by reducing the quality of static resources. Including increasing JPEG and video compression levels.

Note: You generally don’t have to worry about cross-domain resource sharing (CORS) when using a CDN. External resources loaded in HTML aren’t subject to CORS policy: you only have to enable CORS for resources that are loaded via AJAX (see Chapter 15).

18.2 Future-Proofing Your Website

You may be used to hosting them on the same server where all your dynamic HTML is generated. However, if you want to maximize the performance of your site (or allow for doing so in the future), you will want to make it easy to host your static resources on a content delivery network (CDN).

CDN
a server that’s optimized for delivering static resources. It leverages special headers (that we’ll learn about soon) that enable browser caching. Also, CDNs can enable geographic optimization; that is, they can deliver your static content from a server that is geographically closer to your client. Individual time savings may be small, but if you multiply across all of your users, requests, and resources, it adds up fast.

It’s quite easy to “future-proof” your website so that you can move your static content to a CDN when the time comes, and I recommend that you get in the habit of always doing it. What it boils down to is creating an abstraction layer for your static resources so that relocating them all is as easy as flipping a switch.

Most of your static resources will be referenced in HTML views:

  • <link>: CSS files
  • <script>: JavaScript files
  • <img>: images
  • multimedia embedding tags
  • static references in CSS: usually the background-image property
  • static resources are sometimes referenced in JavaScript: code that dynamically changes or inserts <img> tags or the background-image property.

18.2.1 Static Mapping

Strategy for making static resources relocatable, and friendly to caching, is the concept of mapping: what we are concerned with is the logical organization of our static resources, and we'll focus on making it easy to use only this organization when specifying static resources. For example: always write 

<img src="/img/meadowlark_logo.png" alt="Meadowlark Travel Logo">

instead of: 

<img src="//s3-us-west- 2.amazonaws.com/meadowlark/img/meadowlark_logo-3.png" alt="Meadowlark Travel Logo">

Tip: We will be using “protocol-relative URLs” to reference our static resources. This refers to URLs that begin only with / not http:/ or https://. This allows the browser to use whatever protocol is appropriate. If the user is viewing a secure page, it will use HTTPS; otherwise, it will use HTTP. Obviously, your CDN must support HTTPS, and I haven’t found one that doesn’t.

This boils down to a mapping problem:

  • we wish to map less specific paths (/img/meadowlarklogo.png) to more specific paths (//s3-us-west- 2.amazonaws.com/meadowlark/img/meadowlarklogo-3.png).
  • Furthermore, we wish to be able to change that mapping at will. For example, before you sign up for an Amazon S3 account, you may wish to host your images locally (//meadowlarktravel.com/img/meadowlarklogo.png).

    All we’re doing to achieve our mapping is adding something to the start of the path, which we’ll call a base URL. The mapping scheme could be more sophisticated: e.g. we can employ a database of digital assets to map "Meadowlark Logo" to (but NOT recommended, because using filenames and paths is a pretty standard and ubiquitous way to orgnize content, and you should have a compelling reason to deviate from that). A more sophisticated mapping scheme is to employ asset versioning (discussed later), e.g. you could write a mapper that would map /img/meadowlarklogo.png to /img/meadowlarklogo-5.png.

We're going to be sticking with a very simple mapping scheme: just add a base URL - we're assuming all static assets begin with a slash. Since we’ll be using our mapper for several different types of files (views, CSS, and JavaScript), we’ll want to modularize it.

18.2.2 1. Create a file called lib/static.js:

var baseUrl = '';
exports.map = function(name){
    return baseUrl + name;
}

Now it's fine to have our static resources hosted on localhost. We'll probably want to read the value of baseUrl from a configuration file.

Note: Keep your asset mapper as fast as possible, since it is going to be used everywhere. So don't be tempted to add some functionality, e.g. checking for the presence of a beginning slash in the asset name, etc. We can statically analyze our code as part of our QA toolchain to make sure our asset names always start with a slash.

18.7 Serving Static Resources

What is the best way to store the assets? It helps to understand the headers that the browser uses to determine how (and whether) to cache a resource:

  • Expires/ Cache-Control: These two headers tell your browser the maximum amount of time a resource can be cached. They are taken seriously by the browser. It’s important to understand that a browser may remove the image from the cache prematurely, for reasons you have no control over (For example, the user could clear the cache manually, or the browser could clear your resource to make room for other resources the user is visiting more frequently). You onlyneed one of these headers, and Expires is more broadly supported (recommended). If the resource is in the cache, and it has not expired yet, the browser will not issue a GET request at all, which improves performance, especially on mobile.
  • Last-Modified/ ETag: These two tags provide a versioning of sorts: if the browser needs to fetch the resource, it will examine these tags before downloading the content. A GET request is still issued to the server, but if the values returned by these headers satisfy the browser that the resource hasn’t changed, it will not proceed to download the file. As the name implies, Last-Modified allows you to specify the date the resource was last modified. ETag allows you to use an arbitrary string, which is usually a version string or a content hash.

When serving static resources, you should use the Expires header and either Last-Modified or ETag. Express’s built-in static middleware sets Cache-Control, but doesn’t handle either Last-Modified or ETag. So, while it’s suitable for development, it’s not a great solution for deployment.

If you choose to host your static resources on a CDN, such as Amazon CloudFront, Microsoft Azure, or MaxCDN, the advantage is that they will handle most of these details for you. You will be able to fine-tune the details, but the defaults provided by any of these services are already good.

If you don’t want to host your static resources on a CDN, but want something more robust than Express’s built-in connect middleware, consider using a proxy server, such as Nginx (see Chapter 12), which is quite capable.

18.8 Changing Your Static Content

Caching significantly improves the performance of your website, but it isn’t without its consequences. In particular, if you change any of your static resources, clients may not see them until the cached versions expire in your browser. Google recommends you cache for a month, preferably a year. Imagine that!

The solution is finterprinting. Fingerprinting simply decorates the name of the resource with some kind of version information. When you update the asset, the resource name changes, and the browser knows it needs to download it.

Example: Our logo /img/meadowlardlogo.png. If we rename our logo /img/meadowlarklogo-1.png and relect that name change in our HTML, the browser will be forced to download it.

If we have a lot of images on our site, this is where we might consider making your static mapper more sophisticated. For example, we might store the current version of all our digital assets in a database, and the static mapper could look up the asset name.

At the very least, we should fingerprint our CSS and JavaScript files, because they are essential to the functionality of web app.

A popular alternative to fingerprinting individual files is to bundle your resources. Bundling takes all of your CSS and smashes it into one file that’s impossible for a human to read, and does the same for your client-side JavaScript. Since new files are being created anyway, it’s usually easy and common to fingerprint those files.

18.9 Bundling and Minification

In an effort to reduce HTTP requests and reduce the data sent over the wire, “bundling and minification” has become popular.

  • Bundling takes like files (CSS or JavaScript) and bundles multiple files into one (thereby reducing HTTP requests).
  • Minification removes anything unnecessary from your source, such as whitespace (outside of strings), and it can even rename your variables to something shorter.

One additional advantage of bundling and minification is that it reduces the number of assets that need to be fingerprinted.

Still, things are getting complicated quick! Fortunately, there are some Grunt tasks that will help us manage the madness.

18.10 A Note on Third-Party Libraries

You’ll notice I haven’t included jQuery in any bundles in these examples. jQuery is so incredibly ubiquitous, I find that there is dubious value in including it in a bundle: the chances are, your browser probably has a cached copy.

The gray area would be libraries such as Handlebars, Backbone, or Bootstrap: they’re quite popular, but not as likely to be always cached in the browser.

Guildline:

  • If you’re using only one or two third-party libraries, it’s probably not worth bundling them with your scripts.
  • If you’ve got five or more libraries, though, you might see a performance gain by bundling the libraries.

18.12 Summary

For what seems like such a simple thing, static resources are a lot of trouble. However, they probably represent the bulk of the data actually being transferred to your visitors, so spending some time optimizing them will yield substantial payoff.

Whatever technique you choose to use to serve your static resources, I highly recommend hosting them separately, preferably on a CDN. Especially if you spend a little time on your deployment system, so deploying static resources to one location and your application to another is automatic.

If you’re concerned about the hosting costs of CDNs, I encourage you to take a look at what you’re paying now for hosting. Most hosting providers essentially charge for bandwidth, even if you don’t know it. However, if all of a sudden your site is mentioned on Slashdot, and you get “Slashdotted,” you may find yourself with a hosting bill you didn’t expect. CDN hosting is usually set up so that you pay for what you use. To give you an example, a website that I manage for a medium-sized regional company, which uses about 20 GB a month of bandwidth, pays only a few dollars per month to host static resources (and it’s a very media-heavy site).

The performance gains you realize by hosting your static resources on a CDN are significant, and the cost and inconvenience of doing so is minimal, so I highly recommend going this route.

19.1 HTTPS

There is no such things as perfect security. However HTTPs is considered sufficiently secure for banking, corporate security, and healthcare.

HTTPS does not provide authentication, but it lays the groundwork for authentication. Security is as strong as the weakest link, and the first link in that chain is the network protocol.

The HTTPS protocol is based on the server having a public key certificate (sometimes called an SSL certificate). The current standard format for SSL certificate is called X.509.

The idea behind certificates:

  • Certificate authorities (CAs) issue certificates.
  • A CA makes trusted root certificates available to browser vendors.
  • Browsers include these trusted root certificates when you install a browser, and that's what establishes the chain of trust between the CA and the browser. For this chain to work, your server must use a certificate issued by a CA.

HOW does one go about acquiring a certificate?

  • generate your own.
  • get one from a free CA.
  • purchase one from a commercial CA.

19.1.1 Generating Your Own Certificate

Suitable only for development and testing purposes (and possibly for intranet deployment).

Note: If you control the distribution and installation of browsers, you can automatically install your own root certificate when you install the browser. Unless you have a very solid reason to take this approach, it's generally more trouble than it's worth.

To generate your own certificate, you'll need an OpenSSL implementation:

19.1.2 Using a Free Certificate Authority

CACert employs a point-based "web of trust" to ensure you are who you say you are. To get enough points to be issued a certificate, you have to meet with a CACert member who is qualified as an “assurer.” Or you can attend events at which you can get points.

CACert is not currently supported by any major browser. It is likely that they will eventually be supported by Mozilla Firefox it’s unlikely that it will ever be supported for Google Chrome, Internet Explorer, or Apple Safari. It's only recommended for development or testing purposes.

All of the major certificate vendors (such as Comodo and Symantec) offer free trial certificates that last anywhere from 30 to 90 days. This is a valid option if you want to test a commercial certificate.

19.1.3 Purchasing a Certificate

Currently, 90% of the approximately 50 root certificates distributed with every major browser are owned by four companies: Symantec (who purchased VeriSign), Comodo Group, Go Daddy, and GlobalSign.

Purchasing directly from a CA can be quite expensive (> $300/year). A less expensive option is going through a reseller (<$10/year). There is no difference whatsoever in the level of encryption offered between a $10 certificate and a $1,500 certificate.

There are four considerations I use in selecting a certificate vendor:

  • Customer support: Often, your hosting provider will resell certificates, and in my experience, they provide a higher level of customer support.
  • Avoid chained root certificates: It is common to chain certificates, meaning you actually require multiple certificates to establish a secure connection, resulting in additional installation effort. This is another reason to look for good customer support. If you ask whether the root certificate is chained, and they can’t or won’t tell you, you should look elsewhere.
  • Single-domain, multisubdomain, multidomain, and wildcard certificates: Avoid single-domain certificates, though it can be a good option for the extremely budget conscious (you can always set up redirects to funnel requests to the proper domain). Multisubdomain certificates require you to know in advance what subdomains you want to use. You might be better off going with a wildcard certificate, which are generally more expensive. But they will work for any subdomain, and you never have to specify what the subdomains are.
  • Domain, organization, and extended validation certificates: There are three kinds of certificates:
    • domain certificates: simply provide confidence that you’re doing business with the domain that you think you are. Recommended.
    • organization certificates: provide some assurance about the actual organization you’re dealing with.
    • extended validation certificates: They are like organization certificates in that they verify the existence of the organization. Recommended.

Once you’ve purchased a certificate, you will have access to a secure area (over HTTPS) where you can download your private key and the certificate. Avoid certificate vendors who will email you the private key.

Standard extensions for the private key are .pem and sometimes .key. The certificate will have an extension of .crt, .cer, or .der.

19.1.5 A Note on Ports

Most browsers will simply not display the port number if you explicitly specify port 80/443.

There’s no way to run HTTP and HTTPS on the same port (technically, it’s possible, but there’s no good reason to do it, and the implementation would be very complicated).

If you want to run your HTTP app on port 80, or your HTTPS app on port 443, you have two things to consider:

  • First is that many systems already have a default web server running on port 80. For example, if you’re using OS X and you have web sharing enabled, Apache will be running on port 80, and you won’t be able to start your app on port 80.
  • The other thing to know is that on most operating systems, ports 1–1024 require elevated privileges to open. On a Linux or OS X machine, if you attempt to start your app on port 80, it will fail with an EACCES error. To run on port 80 or 443 (or any port under 1025), you’ll need to elevate your privileges by using the sudo command.

Unless you’re managing your own servers, you probably don’t have root access to your hosted account: so what happens when you want to run on port 80 or 443? Generally, hosting providers have some kind of proxy service that runs with elevated privileges that will pass requests through to your app, which is running on a nonprivileged port. We’ll learn more about this in the next section.

19.1.6 HTTPS and Proxies

When you want to scale your site out to handle more traffic, you will want to use a proxy server such as Nginx (see Chapter 12). If your site is running in a shared hosting environment, it is almost certain that there will be a proxy server that will route requests to your application.

The client (the user’s browser) will communicate with the proxy server, not your server. The proxy server, in turn, will most likely communicate with your app over regular HTTP (since your app and the proxy server will be running together on a trusted network). You will often hear people say that the HTTPS terminates at the proxy server.

For the most part, once you or your hosting provider has correctly configured the proxy server to handle HTTPS requests, you won’t need to do any additional work. The exception to that rule is if your application needs to handle both secure and insecure requests.

Three solutions to this problem:

  1. Configure your proxy to redirect all HTTP traffic to HTTPS,

    In essence forcing all communication with your application to be over HTTPS. This approach is becomming much more common, and it is an easy solution.

  2. Somehow communicate the protocol used in the client-proxy communication to the server.

    The usual way to communicate this is through the X-Forwarded-Proto header. For example, to set this header in Nginx: 

    proxy_set_header X-Forwarded-Proto $scheme;

    Then, in your app, you could test to see if the protocol was HTTPS: 

    app.get('/', function(req, res){
    // the following is essentially
    // equivalent to: if(req.secure)  <----
    if(req.headers['x-forwarded-proto']==='https') {
        res.send('line is secure');
    } else {
        res.send('you are insecure!');
    }
});

    CAUTION: In Nginix, there is a separate server configuration block for HTTP and HTTPS. If you fail to set the X-Forwarded-Protocol in the configuration block corresponding to HTTP, you open yourself up to the possibility of a client spoofing the header and thereby fooling your application into thinking that the connection is secure even though it isn’t. If you take this approach, make sure you always set the X-Forwarded-Protocol header.

  3. Express provides some convenience properties that change behavior (quite correctly) when you're using a proxy.

    Don’t forget to tell Express to trust the proxy by using app.enable('trust proxy'). Once you do, req.protocol, req.secure, and req.ip will refer to the client’s connection to the proxy, not to your app.

19.2 Cross-Site Request Forgery

Cross-site request forgery (CSRF) attacks exploit the fact that users generally trust their browser and visit multiple sites in the same session.

In a CSRF attack, script on a malicious site makes requests of another site: if you are logged in on the other site, the malicious site can successfully access secure data from another site.

To prevent CSRF attacks, you must have a way to make sure a request legitimately came from your website.

The way we do this is to pass a unique token to the browser. When the browser then submits a form, the server checks to make sure the token matches.

The csurf middleware will handle the token creation and verification for you; all you’ll have to do is make sure the token is included in requests to the server.

Tip: If you have an API, you probably don’t want the csurf middleware interfering with it. If you want to restrict access to your API from other websites, you should look into the “API key” functionality of connect-rest. To prevent csurf from interfering with your middleware, link it in before you link in csurf.

19.2.1 1. Install the csurf middleware:

npm install --save csurf

19.2.2 2. Link it in and add a token to res.locals:

// this must come after we link in cookie-parser and connect-session

app.use(function(req, res, next){
    res.locals._csrfToken = req.csrfToken();
    next();
});

The csurf middleware adds the csurfToken method to the request object. We don’t have to assign it to res.locals; we could just pass req.csurfToken() explicitly to every view that needs it, but this is generally less work.

19.2.3 3. Add tokens to all the forms:

    Name: <input type="text" name="name"><br>

    <button type="submit">Submit</button>
</form>

19.3 Authentication

Authentication is a big, complicated topic. DON'T TRY TO DO IT YOURSELF!

19.3.1 Authentication vs. Authorization

Often used interchageably, there is a subtle difference:

  • Authentication: refers to verifying user's identities - They are what they say they are.
  • Authorization: refers to determining what a user is authorized to access, modify, or view.

Usaually (but not always), authentication comes first, and then authorization is determined.

The complexity of your authorization system is dependant on the type of application you're writing.

19.3.2 The Problem with Passwords

Every security system is as strong as its weakest link. And passwords require the user to invent a password — and there’s your weakest link. Humans are notoriously bad at coming up with secure passwords.

It’s a problem that you, the app designer, cannot do much to fix. However, there are things you can do that promote more secure passwords.

  • One is to pass the buck and rely on a third party for authentication.
  • The other is to make your login system friendly to password management services, like LastPass, RoboForm, and PasswordBox.

19.3.3 Third-Party Authentication

Third-party authentication takes advantage of the fact that pretty much everyone on the Internet has an account on at least one major service (Google/Facebook/Twitter/LindedIn etc.). All of these services provide a mechanism to authenticate and identify your users through their service.

Third-party authentication / federated authentication / delegated authentication: all these terms are largely interchangeable:

  • Federated authentication: usually associated with Security Assertion Markup Language (SAML) and OpenID
  • Delegated authentication: often associated with OAuth.

Three major advantages:

  • Your authentication burden is lowered.
  • It reduces "password fatigure". (You can use passwords manager such as LastPass to store your hundreds of passwords.)
  • It is "frictionless": it allows your users to start using your site nore quickly, with credentials they already have.

Problem: Most people are using the same passwords for most of those sites which they use that passoword for. If even one of those sites is breached, their password becomes known.

Downsides:

  • There are folks out there who don't have an account on Google, Facebook, etc.
  • Among the people who do have such accounts, suspicion (or a desire for privacy) may make them unwilling to use those credentials to log onto your site.

A more practical solution to this problem: Encouraging users to use an existing account, but those who don't have them (or are unwilling to use them to access your service) can create a new login for your service.

19.3.5 Authentication Versus Registration and the User Experience

Authentication vs. Registration:

  • Authentication refers to verifying a user’s identity, either with a trusted third party, or through credentials you’ve provided the user.
  • Registration is the process by which a user gets an account on your site.

When users join your site for the first time, it should be clear to them that they’re registering.

  • Using a third-party authentication system, we could register them without their knowledge if they successfully authenticate through the third party. This is not generally considered a good practice,
  • and it should be clear to users that they’re registering for your site (whether they’re authenticating through a third party or not), and provide a clear mechanism for canceling their membership.

One user experience situation to consider is “third-party confusion.”: the user may very well have forgotten what registration service was originally used. If you feel that you have a firm grasp on the social networks your users use, you can ease this problem by having a “primary” authentication service. For example: big button for primary authentication service, smaller buttons or even just text links for secondary services.

19.3.6 Passport

Passport is a very popular and robust authentication module for Node/Express. It is not tied to any one authentication mechanism; rather, it is based on the idea of pluggable authentication strategies (including a local strategy if you don’t want to use third-party authentication)

Tip: Using Passport for authentication is a fair amount of work. There are projects such as LockIt that try to provide a more “off the shelf” solution. If you ever need to customize an authentication solution, Passport is a great place to start.

Important: With third-party authentication, your app never receives a password. That is handled entirely by the third party. This is a good thing: it’s putting the burden of secure handling and storage of passwords on the third party.

The whole process relies on redirects. The authentication actually occurs on the third-party website.

The browser can do three things:

  • make an HTTP request
  • display the response
  • perform a redirect

While using Passport, there are four steps that our app will be responsible for (in the "Meadowlark" column):

Figure 18-2: Detailed view of third-party authentication flow

  1. 1. Login page

    In this step, we'll be constructing a request to be sent to a third party (via a redirect).

    In this page, the user can choose the login method. If you’re using a third-party authentication, it’s usually just a button or a link. If you’re using local authentication, it will include username and password fields.

  2. 2. Construct authentication request

    In this step, you’ll be constructing a request to be sent to a third party (via a redirect).

    The details of this request are complicated and specific to the authentication strategy. Passport (and the strategy plugin) will be doing all the heavy lifting here.

    The auth request includes protection against “man in the middle” attacks, as well as other vectors an attacker might exploit. Usually the auth request is short-lived, so you can’t store it and expect to use it later: this helps prevent attacks by limiting the window in which an attacker has time to act.

    This is where you can request additional information from the third-party authorization mechanism. For example, it’s common to request the user’s name, and possibly email address. Keep in mind that the more information you request from users, the less likely they are to authorize your application.

  3. 3. Verify authentication response

    Assuming the user authorized your application, you’ll get back a valid auth response from the third party, which is proof of the user’s identity.

    The details of this validation are complicated and will be handled by Passport (and the strategy plugin).

    Included in the auth response will be an ID for the user that is unique to that specific third party, as well as any additional details you requested in step 2.

    To enable step 4, we must “remember” that the user is authorized. The usual way to do this is to set a session variable containing the user’s ID, indicating that this session has been authorized (cookies can also be used, though I recommend using sessions).

  4. 4. Verify authorization

    In step 3, we stored a user ID in the session. The presence of that user ID allows us to retrieve a user object from the database that contains information about what the user is authorized to do. In this manner, we don’t have to authenticate with the third party for every request. This task is simple, and we no longer need Passport for this: we have our own User object that contains our own authentication rules. (If that object isn’t available, it indicates the request isn’t authorized, and we can redirect to the login or “Not Authorized” page.)

19.3.7 Role-Based Authorization

Example: let’s say we only want customers to see their account views (employees might have an entirely different view where they can see user account information).

Remember that in a single route, you can have multiple functions, which get called in order.

  1. 1. Create a function clled customerOnly that will allow customers:
    function customerOnly(req, res){
    var user = req.session.passport.user;
    if(user && req.role==='customer') return next();
    res.redirect(303, '/unauthorized');
}
  2. 2. Let's also create a employeeOnly function that will operate a little differently:

    Let’s say we have a path sales that we want to be available only to employees. Furthermore, we don’t want nonemployees to even be aware of its existence, even if they stumble on it by accident. So, for a little added security, we want nonemployees to see a /regular 404 page when they visit the /sales page, giving potential attackers nothing to work with: 

    function employeeOnly(req, res, next){
    var user = req.session.passport.user;
    if(user && req.role==='employee') return next();
    next('route');   // <----
}

    Calling next('route') will not simply execute the next handler in the route: it will skip this route altogether. this will eventually pass to the 404 handler, giving us the desired result.

    Here’s how easy it is to put these functions to use: 

    // customer routes
app.get('/account', customerOnly, function(req, res){
    res.render('account');
});
app.get('/account/order-history', customerOnly, function(req, res){
    res.render('account/order-history');
});
app.get('/account/email-prefs', customerOnly, function(req, res){
    res.render('account/email-prefs');
});

// employer routes
app.get('/sales', employeeOnly, function(req, res){
    res.render('sales');
});
  3. 3. Allow multiple roles:

    It should be clear that role-based authorization can be as simple or as complicated as you wish. For example, what if you want to allow multiple roles? 

    function allow(roles) {
    var user = req.session.passport.user;
    if(user && roles.split(',').indexOf(user.role)!==-1) return next();
    res.redirect(303, '/unauthorized');
}
app.get('/account', allow('customer,employee'), function(req, res){
    res.render('account');
});

    You could even authorize on other properties, such as the length of time a user has been a member or how many vacations that user has booked with you.

21.1 The First Principle of Debuggin

Debugging: The process of finding and eliminating defects.

The first and most important principle of debugging is the process of eliminating. – The more you can eliminate, the fewer places you have to look.

Elimination can take many forms, here are some common examples:

  • Systematically commenting out or disabling blocks of code.
  • Writing code that can be covered by unit tests; the unit tests themselves provide a framework for elimination.
  • Analyzing network traffic to determine if the problem is on the client or server side.
  • Testing a different part of the system that has similarities to the first.
  • Using input that has worked before, and changing that input one piece at a time until the problem exhibits.
  • Using version control to go back in time, one step at a time, until the problem disappears.
  • “ Mocking ” functionality to eliminate complex subsystems.

Elimination is not a silver bullet, though. Often, problems are due to complex interactions between two or more components: eliminate (or mock) any one of the components, and the problem could go away, but the problem can’t be isolated to any single component. Even in this situation, though, elimination can help narrow down the problem, even if it doesn’t light up a neon sign over the exact location.

When you consider a component to eliminate, walk through how the removal of that component will affect the system. This will inform you about what to expect and whether or not removing the component tells you anything useful.

21.6.9 Connect middleware (nodemodules/connect)

There is very little middleware in Express now (the static middleware and the router being the notable exceptions). Most of the middleware actually comes from Connect. Express 4.0 no longer bundles Connect.

Connect has also been shedding some of its middleware into standalone packages. Here are the locations of some of the more important ones:

  1. Session middleware (nodemodules/express-session/index.js, function session)

    A lot goes into making sessions work, but the code is pretty straightforward. You may want to set a breakpoint in this function if you’re having issues that are related to sessions. Keep in mind that it is up to you to provide the storage engine for the session middleware.

  2. Logger middleware (nodemoduels/morgan/index.js, function logger)

    This is a debugging aid, not to be debugged itself. However, there’s some subtlety to the way logging works that you’ll get only by stepping through the logger middleware once or twice. The first time I did it, I had a lot of “aha” moments, and found myself using logging more effectively in my applications, so I recommend taking a tour of this middleware at least once.

  3. URLencoded body parsing (nodemodules/body-parser/index.js, function urlencoded)

    The manner in which request bodies are parsed, is not really that complicated, and stepping through this middleware will help you understand the way HTTP requests work. Aside from a learning experience, you won’t find that you need to step into this middleware for debugging very often.

22.1 Domain Registration and Hosting

Usually, when you purchase hosting, you’re automatically assigned a subdomain, which can be thought of as something between a marketing-friendly domain name and an IP address (for example, ec2-54-201-235-192.us-west-2.compute.amazonaws.com).

Once the requests reach your website, it is possible to take action based on the URL that was used. It is more common to redirect from http://meadowlarktravel.com/ to http://www.meadowlarktravel.com/

Recommended is separating domain registration and hosting for reasons of security and flexibility.

22.1.1 Domain Name System

The DNS is what's responsible for mapping domain names to IP addresses.

22.1.2 Security

Domain names are valuable:

  • If a hacker were to completely compromise your hosting service and take control of your hosting, but you retained control of your domain, you could get new hosting and redirect the domain.
  • If, on the other hand, your domain were compromised, you could be in real trouble.

Your reputation is tied to your domain, and good domain names are carefully guarded.

The upshot is that you should take domain security very seriously, perhaps even more seriously than your data (depending on how valuable your data is). I’ve seen people spend inordinate amounts of time and money on hosting security while getting the cheapest, sketchiest domain registration they can find. Don’t make that mistake. (Fortunately, quality domain registration is not particularly expensive.)

Given the importance of protecting ownership of your domain, you should employ good security practices with respect to your domain registration. At the very least, you should use strong, unique passwords, and employ proper password hygiene (no keeping it on a sticky note attached to your monitor).

Preferably, you should use a registrar that offers two-factor authentication. Don’t be afraid to ask your registrar pointed questions about what is required to authorize changes to your account. The registrars I recommend are Name.com and Namecheap.com. Both offer two-factor authentication.

When you register a domain, you must provide a third-party email address that’s associated with that domain you should employ the same security standards as you do with your domain registrar account (good password hygiene and two-factor authentication).

22.1.3 Top-Level Domains

What your domain ends with (such as .com or .net) is called a top-level-domain (TLD). Generally speaking, there are two types of TLD: country code TLDs and general TLDs. Country code TLDs are designed to provide a geographic categorization they are often used for “clever” domains, such as placehold.it and goo.gl.

General TLDs (gTLDs) include the familiar .com, .net, .gov, .fed, .mil, and .edu. While anyone can acquire an available .com or .net domain, there are restrictions in place for the others mentioned. For more information, see Table 21-1.

The Internet Corporation for Assigned Names and Numbers (ICANN) is ultimately responsible for management of TLDs. For the foreseeable future, .com will probably remain the “premium” TLD, and the hardest one to get real estate in.

When picking a domain, you should consider how it’s going to be used.

  • For marketing primarily electronically: focus more on getting a catchy or meaningful domain than a short one.
  • For print advertising or manually entering URLs: consider a shorter domain name.

It's a common practice to have two domains: a short, easy-to-type one, and a longer one more suitable for marketing.

22.1.4 Subdomains

Where a TLD goes after your domain, a subdomain goes before it. By far, the most common subdomain is www.

Recommendation: Using no subdomain for your primary domain – http://meadowlarktravel.com instead of http://www.meadowlardtravel.com. It's shorter and less busy. You can always use redirection to relay traffic without www.

Other urposes of subdomains:

  • blogs., api., m. (for a mobile site). A good proxy, though, can redirect traffic appropriately based on either subdomain or path, so the choice should be more content-focused than technology-focused.
  • Micorsites are also good candidates for subdomains.
  • To separate admin interfaces from public interfaces (admin.meadowlarktravel.com, for employees only).

Recommendation: subdomain can be used to compartmentalize significantly different parts of your website or service – e.g., api.meadowlarktravel.com.

Microsite
sites that have a different appearance than the rest of the site, usually highlighting a single product or subject.

Your domain registrar, unless you specify otherwise, will redirect all traffic to your server regardless of subdomain. It is up to your server (or proxy), then, to take appropriate action based on the subdomain.

22.1.5 Nameservers

Nameserver is what you'll be asked to provide when you establish hosting for your website. Your hosting service will do most of the work for you. What you do is to go to your domain registrar and set the nameservers for the domain you want to host.

  1. Map domain to website directly

    Since the server (or proxy) that actually serves your website has an IP address, we can cut out the middleman by registering that IP address with the DNS resolvers.

    For this approach to work, your hosting service must assign you a static IP address. Commonly, hosting providers will give your server(s) a dynamic IP address, which means it may change without notice.

    If you want to map your domain to your website directly (skipping your host’s nameservers), you will either be adding an A record or a CNAME record. An A record maps a domain name directly to an IP address, whereas a CNAME maps one domain name to another. CNAME records are usually a little less flexible, so A records are generally preferred.

    Domain mapping is usually aggressively cached, meaning that when you change your domain records, it can take up to 48 hours for your domain to be attached to the new server.

    Best practice: If you need something to go live precisely at a certain time, you should not rely on DNS changes. Rather, modify your server to redirect to the “coming soon” site or page, and make the DNS changes in advance of the actual switchover. At the appointed moment, then, you can have your server switch over to the live site, and your visitors will see the change immediately, regardless of where they are in the world.

22.1.6 Hosting

How you select a hosting provider depends very much on your needs.

  1. Traditional hosting, or cloud hosting?

    The term "cloud" is just a fancy way to say "the Internet" or "part of the Internet". Hosting in the cloud usually implies a certain commoditizing of computing resources. The difference is between knowing (and caring) what actual server your app is hosted on, and knowing it's hosted on some server in the cloud, and it could just as easily be moved over to a different one without knowing (or caring).

    Cloud hosting is also highly virtualized. The two terms are becoming synonymous.

    Cloud represents a subtle shift in thinking. All that's changed is that you're more removed from the hardware.

    "Traditional" hosting will eventually disappear altogether.

  2. XaaS

    When considering cloud hosting, you will come across the acronyms SaaS, PaaS, and IaaS:

    • Software as a Service (SaaS): generally describes software (websites, apps) that are provided to you: you just use them. An example would be Google Documents or Dropbox.
    • Platform as a Service (PaaS): provides all of the infrastructure for you (operating systems, networking — all of that is handled). All you have to do is write your applications. While there is often a blurry line between PaaS and IaaS (and you will often find yourself straddling that line as a developer), this is generally the service model we’re discussing in this book. If you’re running a website or web service, PaaS is probably what you’re looking for.
    • Infrastructure as a Service (IaaS): gives you the most flexibility, but at cost. All you get are virtual machines and a basic network connecting them. You are then responsible for installing and maintaining operating systems, databases, and network policies. Unless you need this level of control over your environment, you will generally want to stick with PaaS. (Note that PaaS does allow you to have control over the choice of operating systems and network configuration: you just don’t have to do it yourself.)
  3. The behemoths

    The big-three: Microsoft (Azure), Amazon, and Google. They are all quite good.

    If your hosting needs are modest, there will be minimal price difference among the three. If you have very high bandwidth or storage needs, you will have to evaluate the services more carefully, as the cost difference could be greater.

    Don't overlook Azure. Microsoft has bent over backwards to make it friendly to not just Node, but the open source community. Microsoft offers a one-month Azure trial (definitely recommended). MS also offers Node APIs for all of their major services, including their cloud storage service. Azure offers Git-based deployments, an excellent cloud storage system (with a JavaScript API), as well as good support for MongoDB.

    Downside to Azure: They don't offer a pricing tier for small projects (>80 USD/Month). However, you can easily host multiple projects for that price, so if you're looking to consolidate a bunch of websites, it can be very cost effective.

    Amazon offers the most comprehensive set of resources: SMS, cloud storage, email services, payment services (ecommerce), DNS, and more. Amazon offers a free usage tier.

    Google's cloud platform does not yet offer an option for Node hosting, though Node apps can be hosted through their IaaS service. Google does not currently offer a free tier or trial.

    It's worth considering Joyent. Nodejitsu (a Joyent partner) provides Node-specific hosting and are experts in the field. They offer a unique option for deployment: a private npm repository. If Git-based deployment doesn't appeal to you, you're encouraged to look into Nodejitsu's npm-based deployment.

  4. Boutique hosting

    These are smaller hosting services. Because they can't compete in terms of infrastructure, instead they usually focus on customer service and support.

    The writer recommends WebFaction for personal projects. Their service is extremely affordable, and they have offered Node hosting for some time now. Remember to ask your hosting providers if they offer (or plan on offering) Node hosting.

22.1.7 Deployment

Don't use FTP to deploy apps. Not only are all your files transmitted unencrypted, your username and password are also. At minimium, you should be using SFTP or FTPS, but there's even a better way: Git-based deployment.

Idea: you use Git for version control anyway, and deployment is essentially a problem of versioning, making Git a natural fit. This techique is not restricted to Git, you could use Mercurial or Subversion for deployment if you want to.

Your development repositories need some way of synchronizing with the deployment repositories. The easiest by far is to use an Internet service like GitHub. You can upgrade to a private Git repository for a fee, or use Atlassian Bitbucket free private repository hosting for up to five users.

While Git-based deployment can be set up on almost any service, Azure offers it out of the box. We'll start with that excellent model, then cover how we can partially emulate this with other hosting providers.

  1. Git deployment

    Git’s greatest strength (and greatest weakness) is its flexibility. It can be adapted to almost any workflow imaginable.

    Tip: We’ve already discussed the value of keeping your binary assets (multimedia and documents) separate from your code repository. Git-based deployment offers another incentive for this approach. If you have four gigabytes of multimedia data in your repository, they’re going to take forever to clone, and you have an unnecessary copy of all of your data for every production server.

    For the sake of deployment, I recommend creating one or more branches specifically for deployment. For example, you might have a production branch and a staging branch. How you use those branches is very much up to your individual workflow.

    One popular approach is to flow from master to staging to production. So once some changes on master are ready to go live, you could merge them into staging. Once they have been approved on the staging server, you could then merge staging into master. But this approach creates the clutter (merges everywhere), and if you have lots of features that need to be staged and pushed to production in different orders, this can get messy quickly.

    A better approach is to merge master into staging and, when you're ready to go live with changes, then merge master into production. In this way, staging and production become less associated: you can even have multiple staging branches to experiment with different features before going live (and you can merge things other than master into them). Only when something has been approved for production do you merge it into production.

    1. What happens when you need to roll back changes?

      This is where things can get complicated. There are multiple techiniques for undoing changes, such as applying the inverse of a commit to undo prior commits (get revert), but not only are these techniques complicated, they can also cause problems down the line.

      A better approach (the writer recommends it) is to treat production (and your staging branches, if you wish) as disposable: they are really just reflections of your master branch at different points in time. If you need to roll back changes, then you just do a git reset --hard <old commit id> on your production branch, and then git push origin production --force. In essence, this is "rewriting history", which is often disencoraged as dangerous or "advanced". While it certainly can be, in this case, it is understood that production is a read-only branch – developers should never commit to it (which is where rewriting history can get you into trouble).

      In the end, it is up to you and your team to decide on a Git workflow. More important than the workflow you pick is the consistency with which you use it, and the training and communication surrounding it.

  2. Deployment to Azure

    With Azure, you can deploy from a GitHub or Bitbucket repository, or a local repository. I strongly recommend you use either GitHub or Bitbucket; it will make it much easier to add people to your development team.

    One important note is that Azure expects your main application file to be called server.js.

    1. Log into your Azure portal, create a new website:
      1. Click the Website icon along the left.
      2. Click New along the bottom.
      3. Choose Quick Create; choose a name and a region, and click Create Web Site.
    2. Set up source control deployment:
      1. Click your website in the main portal window.
      2. Under the “Your site has been created!” message, look for “Set up deployment from source control”; click that link.
      3. Choose either GitHub or Bitbucket; if this is your first time, you will be asked to authorize Azure access to your GitHub or Bitbucket account.
      4. Choose the repository you want to use and the branch (I recommend production).

    If Azure detects that there is an update to the production branch, it will automatically update the code on the server.

    Even better? If you’ve added any new dependencies to package.json, Azure will automatically install them for you. It will also handle file deletions (unsurprisingly, since this is standard Git behavior). In other words, you have seamless development.

    This technique also works well if you scale out your app. So if you have four server instances going, they will all be updated simultaneously with a single push to the appropriate branch.

    If you go to the Azure control panel for your website, you will see a tab labeled Deployments. In that tab, you will see information about the history of deployments, which can be helpful in debugging if something goes wrong with your automatic deployment system. Also, you can redeploy previous deployments, which can be a quick way to revert if there’s a problem.

  3. Manual Git-based deployment

    If your hosting service does not support any kind of automated Git-based deployment, your approach will involve additional steps.

    Our setup is the same:

    • we’re using GitHub or Bitbucket for version control,
    • and we have a branch called production, which is what we want reflected on the production servers.

    For each server, you will have to clone the repository, check out the production branch, and then set up the infrastructure necessary to start/restart your app (which will be dependent on your choice of platform). When you update the production branch, you will have to go to each server, run git pull --ff-only , run npm install (if you’ve updated any dependencies), and then restart the app.

    If your deployments aren’t often, and you don’t have very many servers, this may not represent a terrible hardship, but if you’re updating more often, this will get old fast, and you’ll want to find some way to automate the system.

    Unfortunately, automation is not a simple matter. Git has hooks that allow you to take automated action, but not if it’s a remote repository that’s being updated. If you’re looking for automated deployment, the easiest approach is to run an automated job that runs git pull --ff-only periodically. If an update occurs, you can then run npm install and restart the app.

    Tip: The --ff-only argument to git pull allows only fast-forward pulls, preventing automatic merging or rebasing. If you know the pull is fast-forward only, you may safely omit it, but if you get in the habit of doing it, you will never accidentally invoke a merge or rebase!

  4. Amazon deployment with Elastic Beanstalk

    You can use their product Elastic Beanstalk (EB) to do automated deployment with Git. But it comes with increased complexity. You can find instructions for the various ways to configure EB on the EB documentation page.

22.2 Conclusion

Deployment is no easy job.

Best practice: No matter what deployment strategy you choose, the most important thing you can do is to start production deployments early, before the site is ready to go live. You don’t have to hook up the domain, so the public doesn’t need to know. If you’ve already deployed the site to production servers half a dozen times before the day of launch, your chances of a successful launch will be much higher. Ideally, your functioning website will already be running on the production server long before launch: all you have to do is flip the switch from the old site to the new site.

23.1 The Principles of Maintenance

23.1.1 Have a Longevity Plan

What clients do not appreciate is the unexpected: being told after three years that their site has to be rebuilt when they had an unspoken expectation that it would last five.

Setting expectations about website longevity is part art, part salesmanship, and part science:

  • The science of it involves something that all scientists, but very few web developers, do: keep records. Imagine if you had a record of every website your team had ever launched, the history of maintenance requests and failures, the technologies used, and how long before each site was rebuilt. You may find that certain development approaches work better for your team, or certain platforms or technologies. What I almost guarantee you will find is a correlation between “procrastination” and defects: the longer you put off an infrastructure update or platform upgrade that’s causing pain, the worse it will be. Having a good issue tracking system and keeping meticulous records will allow you to give your client a much better (and more realistic) picture of what the life cycle of their project is going to be.
  • The salesmanship of it boils down to money. Seven years is the maximum realistic life expectancy for websites that have any hope of continuing to be useful.
    • With the clients who have a lot of money, don’t just take their money because they have it: use that extra money to give them something extraordinary.
    • With the clients on a tight budget, you will have to find creative ways to design their website for greater longevity in the face of constantly changing technology.

The art of the matter:

  • understanding what the client can afford, and where you can honestly convince the client to spend more money so they get value where they need it.
  • understanding technology futures, and being able to predict what technologies will be painfully obsolete in five years and which will be going strong.

There’s no way to predict anything with absolute certainty, of course. None of this uncertainty should deter you from having a plan, however: better to have a plan that goes awry than to always be rudderless.

JavaScript and Node are technologies that are going to be around for a while. The Node community is vibrant and enthusiastic, and wisely based on a language that has clearly won. Most important, perhaps, is that JavaScript is a multiparadigm language: object-oriented, functional, procedural, synchronous, asynchronous — it’s all there. This makes JavaScript an inviting platform for developers from many different backgrounds, and is in large part responsible for the pace of innovation in the JavaScript ecosystem.

23.1.9 Prioritize Lead Tracking

In the Internet world, the strongest signal your visitors can give you that they are interested in your product or service is to give you contact information: you should treat this information with the utmost care.

Any form that collects an email or phone number should be tested routinely as part of your QA checklist, and there should always be redundancy when you collect that information. The worst thing you can do to a potential customer is collect contact information and then lose it.

Because lead tracking is so critical to the success of your website, I recommend these five principles for collecting information:

  1. Have a fallback in case JavaScript fails

    If JavaScript should fail for any reason, the form submission should work anyway.

    A great way to test this is to disable JavaScript and use your form.

    The point is that user data is not lost. To implement this, always have a valid and working action parameter in your <form> tag, even if you normally use AJAX.

  2. If you use AJAX, get the URL from the form’s action parameter

    While not strictly necessary, this helps prevent you from accidentally forgetting the action parameter on your <form> tags.

    For example, your form tag could be 

    <form action="submitemail" method="POST">

    Then in your AJAX handler, you would do the following: 

    $('form').on('submit', function(evt) {
    evt.preventDefault();
    var action = $(this).attr(\'action'); 
    /* perform AJAX submission */ 
});
  3. Provide at least one level of redundancy

    A common way to provide redundancy is to send an email in addition to storing the lead. If you take this approach, you should not use a person’s email address, but a shared email address (such as dev@meadowlarktravel.com).

    You could also store the lead in a backup database, or even a CSV file.

    However, whenever your primary storage fails, there should be some mechanism to alert you of the failure. Collecting a redundant backup is the first half of the battle: being aware of failures and taking appropriate action is the second half.

  4. In case of total storage failure, inform the user

    If all of these redundancies fail, the user should receive a message that says something like “We’re sorry, we’re experiencing technical difficulties. Please try again later, or contact support@meadowlarktravel.com.”

  5. Check for positive confirmation, not absence of an error

    It’s quite common to have your AJAX handler return an object with an err property in the case of failure; the client code then has something that looks like this: 

    if(data.err){ 
    /* inform user of failure */ 
} else {
    /*thank user for successful submission */ 
}

    Avoid this approach.

    There’s nothing wrong with setting an err property, but if there’s an error in your AJAX handler, leading the server to respond with a 500 response code or a response that isn’t valid JSON, this approach will fail silently. The user’s lead will disappear into the void, and they will be none the wiser. [ME: literally understood? - check for positive confirmation, not absence of an error]

    Instead, provide a success property for successful submission (even if the primary storage failed: if the user’s information was recorded by something, you may return success). Then your client-side code becomes 

    if(data.success) { 
    /* thank user for successful submission */ 
} else { 
    /* inform user of failure */ 
}

I see it all the time: because developers are in a hurry, they record errors in ways that never get checked.

The end result is the same: your website has quality problems that are going unnoticed. The number one defense you can have against this problem is to provide an easy, standard method for logging errors. Make sure every developer that touches your project is aware of it. It can be as simple as exposing a meadowlarkLog function ( log is often used by other packages).

It also allows you to improve your logging mechanism (for example, flat files are less useful when you scale out your server, so you would modify your meadowlarkLog function to record to a database instead). 23.2 Code Reuse and Refactoring

One tragedy I see all the time is the reinvention of the wheel, over and over and over again.

Usually it’s just small things. All of those little rewritten snippets add up. Worse, it flies in the face of good QA: you’re probably not going to go to the trouble to write tests for all these little snippets. Each snippet — doing the same thing — can have different bugs. It’s a bad habit.

Development in Node and Express offers some great ways to combat this problem.

  • Node brought namespacing (via modules) and packages (via npm),
  • and Express brings the concept of middleware (via Connect).

With these tools at your disposal, it makes it a lot easier to develop reusable code.

23.2.1 Private npm Registry

npm registries are a great place to store shared code. In addition to simple storage, you get versioning, and a convenient way to include those packages in other projects.

There's a little problem: unless you’re working in a completely open source organization, you may not want to create npm packages for all of your reusable code. Besides of the intellectual property protection, your packages could be so organization- or project-specific that it doesn't make sense to make them available on a public registry.

The biggest hurdle to creating your own private registry is that npm currently doesn’t allow you to pull from multiple repositories. So if your package.json file contains a mix of packages from the public npm registry (and it will) and packages from a private registry, npm will fail. The npm team has said they don’t have the resources to implement this feature, but there are alternatives.

One way to handle that problem is to replicate the entire public npm. A better approach is to provide a proxy to the public npm, which will pass requests for a public package through to the public registry, while serving private packages from its own database. Fortunately, there is just such a project: Sinopia.

Sinopia is incredibly easy to install and, in addition to supporting private packages, provides a handy cache of packages for your organization. Since it uses the local filesystem to store private packages, you would definitely want to add the package directory to your backup plan! Sinopia suggests using the prefix test- for local packages: if you’re creating a private registry for your organization, I recommend you use the organization’s name ( meadowlark-).

Since npm is configured to support only one registry, once you “switch over” to using Sinopia (using npm set registry and npm adduser ), you will be unable to use the public npm registry (except through Sinopia). To switch back to using the public npm registry, you can either use npm set registry https://registry.npmjs.org/ , or simply delete the file ~/.npmjs. You will have to do this if you want to publish packages to the public registry.

A much easier solution is to use a hosted private repository. Nodejitsu and Gemfury both offer private npm repositories. Unfortunately, both these services are rather expensive.

23.2.2 Middleware

Writing middleware is not some big, scary, complicated thing. The next step, then, is to put reusable middleware in a package and put it in a npm registry.

We should consider refactoring the middleware to be configured for more general use. Remember that you can pass configuration objects into middleware to make them useful in a whole range of situations. Here is an overview of the most common ways to expose middleware in a Node module. All of the following assume that you’re using exporting these modules as a package, and that package is called meadowlark-stuff:

  1. Module exposes middleware function directly

    Use this method if your middleware doesn’t need a configuration object: 

    module.exports = function(req, res, next){
    // your middleware goes here...remember to call next()
    // or next('route') unless this middleware is expected
    // to be an endpoint
    next();
}

    To use this middlware: 

    var stuff = require('meadowlark-stuff');

app.use(stuff);
  2. Module exposes a function that returns middleware

    Use this method if your middleware needs a configuration object or other information: 

    module.exports = function(config){
    // it's common to create the config object
    // if it wasn't passed in:
    if(!config) config = {};

    return function(req, res, next){
        // your middleware goes here...remember to call next()
        // or next('route') unless this middleware is expected
        // to be an endpoint
        next();
    }
}

    To use this middleware: 

    var stuff = require('meadowlark-stuff')({ option: 'my choice' });

app.use(stuff);
  3. Module exposes an object that contains middleware

    Use this option if you want to expose multiple related middleware: 

    module.exports = function(config){
    // it's common to create the config object
    // if it wasn't passed in:
    if(!config) config = {};

    return {
        m1: function(req, res, next){
            // your middleware goes here...remember to call next()
            // or next('route') unless this middleware is expected
            // to be an endpoint
            next();
        },
        m2: function(req, res, next){
            next();
        }
    }
}

    To use this middleware: 

    var stuff = require('meadowlark-stuff')({ option: 'my choice' });

app.use(stuff.m1);
app.use(stuff.m2);
  4. Module exposes an object constructor

    This is probably the most uncommon method for returning middleware, but is useful if your middleware is well suited for an object-oriented implementation. It is also the trickiest way to implement middleware, because if you expose your middleware as instance methods, they will not be invoked against the object instance by Express, so this will not be what you expect it to be. If you need to access instance properties, see m2 : 

    function Stuff(config){
    this.config = config || {};
}
Stuff.prototype.m1 = function(req, res, next){
    // BEWARE: 'this' will not be what you expect; don't use it
    next();
};
Stuff.prototype.m2 = function(){
    // we use Function.prototype.bind to associate this instance
    // to the 'this property
    return (function(req, res, next){
        // 'this' will now be the Stuff instance   /// <---
        next();
    }).bind(this);   /// <---
);

module.exports = Stuff;

    To use this middleware: 

    var Stuff = require('meadowlark-stuff');

var stuff = new Stuff({ option: 'my choice' });

app.use(stuff.m1);
app.use(stuff.m2());

    Note that we can link in the m1 middleware directly, but we have to invoke m2 (which then returns middleware that we can link in).

23.3 Conclusion

When you’re building a website, the focus is often on the launch. However, a client that is delighted by a newly launched website will quickly become a dissatisfied customer if care isn’t taken in maintaining the website. Approaching your maintenance plan with the same care with which you launch websites will provide the kind of experience that keeps clients coming back.

24.3.1 Be an informed SO user

  • SO tour
  • "How do I ask a good question?"
  • help documentation

24.3.2 Don’t ask questions that have already been answered

If you ask a question that has an easily found answer already on SO, your question will quickly be closed as a duplicate, and people will often downvote you for this, negatively affecting your reputation.

24.3.3 Don’t ask people to write your code for you

You will quickly find your question downvoted and closed if you simply ask “How do I do X?” The SO community expects you to make an effort to solve your own problem before resorting to SO. Describe in your question what you’ve tried and why it isn’t working.

24.3.4 Ask one question at a time

24.3.5 Craft a minimal example of your issue

Crafting a minimal example is good for your debugging skills and for your critical thinking ability, and makes you a good SO citizen.

24.3.6 Learn Markdown

Stack Overflow uses Markdown for formatting questions and answers.

24.3.7 Accept and upvote answers

If multiple people provide acceptable answers, you should pick the one you think is best and accept that, and upvote anyone else you feel offered a useful answer.

24.3.8 If you figure out your own problem before someone else does, answer your own question

SO is a community resource: if you have a problem, chances are, someone else has it too. If you’ve figured it out, go ahead and answer your own question for the benefit of others.

24.4 Contributing to Express

Express and Connect are open source projects, so anyone can submit “pull requests” (GitHub lingo for changes you’ve made that you would like included in the project). This is not easy to do: the developers working on these projects are pros and the ultimate authority on their own projects.

The actual process of contributing is pretty easy: you fork the project in your own GitHub account, clone that fork, make your changes, push them back to GitHub, and then create a pull request, which will be reviewed by someone on the project. If your submissions are small, or are bug fixes, you may have luck simply submitting the pull request. If you’re trying to do something major, you should communicate with one of the main developers and discuss your contribution.

The other way to contribute (indirectly) to the development of Express and Connect is to publish npm packages, specifically middleware. Publishing your own middleware requires approval from no one, but that doesn’t mean you should carelessly clutter the npm registry with low-quality middleware: plan, test, implement, and document, and your middleware will enjoy more success.

If you do publish your own packages, here are the minimum things you should have:

24.4.1 Package name

Unlike GitHub, npm packages are not namespaced by account, so you’re competing globally for names. If you’re writing middleware, it’s customary to prefix your package name with connect- or express- . Catchy package names that don’t have any particular relation to what it does are fine, but even better is a package name that hints at what it does (a great example of a catchy but appropriate package name is zombie, for headless browser emulation).

24.4.2 Package description

Your package description should be short, concise, and descriptive. This is one of the primary fields that is indexed when people search for packages, so it’s best to be descriptive, not clever (there’s room for some cleverness and humor in your documentation, don’t worry).

24.4.3 Author/contributors

24.4.4 License(s)

This is often neglected, and there is nothing more frustrating than encountering a package without a license (leaving you unsure if you can use it in your project).

Available licences:

  • The MIT license is an easy choice if you don’t want any restrictions on how your code is used.
  • If you want it to be open source (and stay open source), another popular choice is the GPL license.

For maximum coverage, dual-license with MIT and GPL.

It’s also wise to include license files in the root directory of your project (they should start with LICENSE).

For an example of this in package.json and in LICENSE files, see my connect-bundle package.

24.4.5 Version

For the versioning system to work, you need to version your packages. Note that npm versioning is separate from commit numbers in your repository: you can update your repository all you like, but it won’t change what people get when they use npm to install your package. You need to increment your version number and re-publish for changes to be reflected in the npm registry.

24.4.6 Dependencies

Be conservative about dependencies in your packages. At a minimum, you should make sure you aren’t listing dependencies that you don’t need.

24.4.7 Keywords

Along with description, keywords are the other major metadata used for people trying to find your package.

24.4.8 Repository

You should have one. GitHub is the most common, but others are welcome.

24.4.9 README.md

The standard documentation format for both GitHub and npm is Markdown. At a minimum, you should describe basic usage (with examples). Even better is to have all options documented. Describing how to run tests goes the extra mile.

24.4.10 Publish your package

The process is quite easy. Register for a free npm account, then follow these steps:

  1. Type npm adduser, and log in with your npm credentials.
  2. Type npm publish to publish your package.

You’ll probably want to create a project from scratch, and test your package by using npm install.
